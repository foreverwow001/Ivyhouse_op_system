# Ivyhouse OP System 專案治理規則

本檔是 Ivyhouse OP System 的有效專案規則來源，用於約束 Phase 0 之後的規格、計畫、實作、審查與交付方式。Ivyhouse OP System 是烘焙營運管理系統，服務對象是內部營運、門市、採購、生產、物流與財務團隊，不是展示型網站或單一報表工具。

---

## 1. 專案定位與規則優先順序

- 專案目標是建立一套可支援門市營運、產品管理、配方管理、原料管理、採購、庫存、生產排程、製作執行、訂單、出貨、對帳與營運分析的業務系統。
- 本專案的核心風險不在頁面呈現，而在主資料治理、流程狀態流轉、角色權限、跨模組共享鍵、財務與對帳一致性。
- 所有對話、規格、計畫、日誌、註解、操作說明、審查結論與專案文件一律使用繁體中文。資料表、欄位、程式語言保留的技術名詞可使用既有英文名稱，但說明文字必須是繁體中文。
- 規則優先順序如下：
  1. `project_rules.md`
  2. 已採納的 `doc/architecture/` 內權威文件
  3. `doc/implementation_plan_index.md` 與對應 plan/log artifact
  4. 角色審查文件與一般討論紀錄
- 若不同文件互相衝突，以較高層級規則為準；若同層級文件衝突，先停下來補決策或澄清，不得憑主觀選一份繼續做。

## 2. 工作模式與變更分級

- 正式流程適用於高風險、跨模組、改動共享契約、涉及資料結構、權限、流程狀態、財務邏輯、部署基線或外部整合的工作。
- 輕量流程只適用於低風險且影響範圍明確的小幅文件修訂、文案修正、排版調整、註解澄清，或單一模組內不改契約的微調。
- 即使走輕量流程，仍必須遵守本檔規則，不得因為變更小就跳過命名、資料邊界、審查與資安要求。
- 若需要向使用者確認範圍、選項、決策或阻斷點，預設優先使用 `vscode_askQuestions`；只有問題無法結構化或該工具不可用時，才可退回一般文字確認。
- 若是在 `/dev` formal workflow 內執行 user-facing gate，仍以 `.github/workflow-core/AGENT_ENTRY.md` 為準：必須先實際呼叫 `vscode_askQuestions`，若確認不可用則 fail-closed，不得退回一般文字確認。
- 任何變更只要命中以下任一條件，必須升級為正式流程：
  - 影響兩個以上模組的共享鍵、共享資料或狀態定義。
  - 涉及 schema、migration、資料回填、批次修正或庫存數量計算。
  - 涉及角色權限、人工覆核責任或登入驗證邏輯。
  - 涉及金額、成本、發票、付款、核銷、對帳或結帳結果。
  - 涉及不可逆操作、批次匯入匯出、外部整合或稽核紀錄。
- 連續兩次修正失敗後，不得盲修；必須回到規格、權威文件與問題證據重新檢查，再決定是否調整方案。

## 3. 架構策略

### 3.1 Repo 形態與系統架構

- Repo 形態採 monorepo，集中管理後端、前端、資料模型、文件與 workflow artifact。
- 系統架構初稿採模組化單體，先以單一業務系統維持一致交易與審計能力，再透過模組邊界與非同步作業處理擴充需求。
- 模組切分以業務邊界與核心流程為主，不以畫面或資料表任意拆分。

### 3.2 模組切分基準

Phase 0 的模組地圖以以下邊界為準：

1. 主資料與參照資料：門市、倉庫、供應商、產品、配方、原料、單位、稅別、客戶類型。
2. 產品與配方：產品定義、配方版本、產出率、替代料、有效期間。
3. 採購與供應：請購、採購單、到貨、驗收、供應商條件。
4. 庫存與批次：庫存台帳、批號、效期、調整、盤點、轉倉。
5. 生產排程與製作執行：生產計畫、工單、領料、回報、報廢、入庫。
6. 訂單與出貨：門市訂單、客戶訂單、揀貨、出貨、退貨。
7. 財務與對帳：發票、收款、付款、核銷、成本歸集、差異分析。
8. 權限、稽核與營運分析：RBAC、操作紀錄、營運報表、例外追蹤。

### 3.3 主資料與共享資料治理策略

- 主資料只能由對應 owner 模組維護，其他模組只能讀取、引用或透過明確服務介面提出變更，不得直接寫入他模組主資料表。
- 每個主資料實體至少要同時定義：內部主鍵、對外業務代碼、中文名稱、狀態、有效期間、建立來源與停用規則。
- 共享鍵治理規則如下：
  - 內部關聯優先使用不可變 surrogate key。
  - 對外顯示與人工對帳使用 business code，但 business code 不可作為跨模組唯一關聯依據。
  - 跨模組契約必須明確標示 key owner、產生規則、不可變欄位與允許更新欄位。
- 禁止跨模組直接寫入共享表；若短期內因單體架構需要共用資料庫，也只能透過 owner module 的 domain service 或受控 repository 存取。
- 主資料、共享鍵、流程狀態與 RBAC 一旦進入正式使用，不得只在聊天紀錄中定義，必須落到 `doc/architecture/` 內的權威文件。

### 3.4 模組邊界規則

- 允許跨模組依賴的前提是：引用的是已公布的共享鍵、狀態字典、查詢介面或事件契約，而不是直接依賴他模組私有資料結構。
- 禁止以下直接耦合：
  - 前端直接推導後端業務狀態，不經後端狀態模型。
  - 模組 A 直接操作模組 B 的 data access 層。
  - 財務模組直接修改生產或庫存來源單據的原始數量以達成對帳。
- 共用契約、主資料定義、流程狀態與架構決策的 repo-native 位置如下：
  - 架構總覽：`doc/architecture/README.md`
  - 模組邊界：`doc/architecture/modules/README.md`
  - 主資料與共享鍵：`doc/architecture/data/README.md`
  - 流程與狀態：`doc/architecture/flows/README.md`
  - 架構決策：`doc/architecture/decisions/README.md`

### 3.5 命名與語彙權威來源

- 在正式 glossary 文件建立前，專案術語以本檔與 `doc/architecture/data/README.md` 為暫定但可執行的權威來源。
- 必須優先使用以下語彙：
  - 產品：可被銷售或生產的成品或半成品。
  - 配方：產品的製作結構與用量定義，必須版本化。
  - 原料：採購與庫存管理的基礎物料。
  - 批次：同一收貨、生產或追溯單位下的可追蹤庫存群組。
  - 工單：生產執行與回報的最小工作單位。
  - 對帳：將業務單據、金流與帳務結果對齊的核對程序。
- 命名衝突時，以主資料 owner 定義為準，不得讓不同模組各自重新命名同一概念。

## 4. 技術基線

以下為 Phase 0 可直接開工的技術初稿；若後續調整，必須在架構決策文件記錄理由與影響。

- 後端：TypeScript + NestJS，採模組化 REST API 為主，必要時以背景工作處理排程、批次與整批對帳。
- 前端／Client：Next.js + React + TypeScript，先以內部後台管理介面為主，桌面優先並兼顧平板操作。
- 資料庫：PostgreSQL 16，作為交易資料、主資料與審計資料的主要來源。
- ORM／查詢層：Prisma 作為主要 schema 與交易操作工具；複雜報表、成本分析與對帳查詢可搭配受控 SQL。
- Migration 工具：Prisma Migrate；所有 migration 必須可重複執行、可審查，涉及資料回填時需附回填與回滾策略。
- 部署：容器化部署。開發環境以 Docker Compose 為主，正式環境暫定採 Cloud Run 搭配託管 PostgreSQL。此決策屬暫定但可開工，若雲平台改變，容器、環境變數與遷移流程不變。
- 可觀測性：結構化 JSON logging、request id、作業 id、審計日誌、OpenTelemetry traces；錯誤追蹤暫定採 Sentry，指標與追蹤暫定採 Prometheus / Grafana 或等價託管服務。

### 資料庫設計原則

- 交易資料必須保留來源單據與狀態，不能只保留最終聚合結果。
- 配方、成本與權限等關鍵規則變更必須版本化，不得覆寫歷史事實。
- 庫存數量以台帳思維管理，任何調整都要保留原因、操作者與對應單據。
- 發票、付款、核銷與成本資料必須可追溯到來源單據與計算基礎。

## 5. Phase 0 權威文件與交付契約

### 5.1 Phase 0 authoritative docs 路徑

Phase 0 期間以下路徑是目前有效的權威文件集合：

1. `project_rules.md`：治理規則、架構基線、技術方向、風險定義。
2. `doc/implementation_plan_index.md`：Phase 0 與後續實作工作的排序、狀態與依賴入口。
3. `doc/architecture/README.md`：架構文件總入口與責任說明。
4. `doc/architecture/modules/README.md`：模組邊界與依賴方向的暫定權威位置。
5. `doc/architecture/data/README.md`：主資料、共享鍵、schema 契約的暫定權威位置。
6. `doc/architecture/flows/README.md`：核心流程、狀態流轉、例外路徑的暫定權威位置。
7. `doc/architecture/decisions/README.md`：技術與架構決策的暫定權威位置。
8. `.github/workflow-core/roles/domain_expert.md`：領域審查方式與 advisory review 觸發原則。

### 5.2 Plan / Log 契約

- `doc/implementation_plan_index.md` 是所有 work unit 的總索引；未先進入 index 的工作不得直接啟動。
- 個別 work unit 一旦從規劃進入執行，必須建立對應的 `doc/plans/Idx-NNN_plan.md` 與 `doc/logs/Idx-NNN_log.md`。
- `depends_on` 只記錄直接阻斷啟動或完成的上游任務，不列所有關聯。
- Plan 必須至少說明：問題範圍、涉及模組、風險面、審查需求、驗收方式、需引用的權威文件。
- Log 必須至少記錄：重要決策、實作範圍、測試結果、未解風險、是否需要 handoff。

### 5.3 Phase 0 前置檢查

在開始任何正式功能實作前，至少要具備以下 Phase 0 前置條件：

1. 已確認 authoritative docs inventory 與文件責任邊界。
2. 已定義初版模組地圖與依賴方向。
3. 已建立初版主資料字典，至少涵蓋門市、產品、配方、原料、供應商、倉庫、工單、訂單、發票、付款等核心實體。
4. 已建立核心流程與狀態模型，至少涵蓋採購到入庫、生產排程到完工、訂單到出貨、發票到對帳。
5. 已建立 RBAC 初版矩陣，定義角色、可見性、操作權限與覆核責任。
6. 已列出共享鍵與跨模組契約，明確指定 key owner 與引用規則。
7. 已確認技術基線與 bootstrap 方向足以支援前述模型。

Phase 0 的退出條件是上述文件皆有可執行初版，且至少完成一次 domain review；若涉及高風險面，需同步完成 security review 或財務審查。

## 6. 高風險面與審查觸發

### 6.1 高風險面定義

以下議題一律視為高風險面：

- 認證、授權、角色權限、資料可見性。
- 主資料定義、共享鍵、跨模組 schema、migration、資料回填。
- 配方版本、用量換算、單位換算、產出率、報廢與替代料。
- 庫存台帳、批次追溯、效期、盤點、轉倉、負庫存處理。
- 生產工單狀態、完工回報、取消與重作邏輯。
- 訂單、出貨、退貨、發票、付款、核銷、對帳、成本歸集。
- 匯入匯出、批次作業、外部整合、Webhook、排程。
- 稽核日誌、個資、供應商與客戶敏感資料、環境密鑰。

### 6.2 Security Review 觸發原則

只要命中以下任一條件，必須進行 security review：

1. 新增或修改登入、session、密碼、權杖、角色權限或 API 保護機制。
2. 新增檔案上傳、批次匯入匯出、外部回呼、第三方整合或背景工作。
3. 新增原始 SQL、migration、資料回填腳本，或能大量讀寫敏感資料的管理操作。
4. 涉及發票、付款、對帳、匯款資訊、個資、營業機密或審計日誌。
5. 涉及不可逆操作，但沒有雙人覆核、明確授權或審計紀錄。

### 6.3 Domain Review / Expert Review 原則

- 凡是碰到主資料字典、schema / migration、workflow / state transition、role / permission / auth、invoice / payment / reconciliation / costing 的變更，都應觸發 domain expert review。
- Domain review 的作用是補足業務邊界與一致性檢查，屬 advisory review，但在 Phase 0 與高風險變更中視為必經審查步驟。
- 若審查發現缺少權威文件，應先補文件或補決策，不得直接繼續實作。

## 7. 專案特定規範

### 7.1 主資料與單位治理

- 產品、配方、原料、供應商、門市、倉庫與客戶都是主資料，不得在交易流程中臨時自由輸入成新的有效實體。
- 每個原料與產品都必須有主要管理單位；若存在採購單位、生產單位、銷售單位不一致，必須定義換算規則與有效期間。
- 配方與成本計算不可依賴自由文字原料名稱，必須引用正式原料鍵。

### 7.2 配方與生產規則

- 配方必須版本化，已被工單引用的配方版本不得直接覆寫，只能停用或建立新版本。
- 生產工單一旦進入已開工或已領料狀態，不可無痕改料或改量；如需修正，必須留下調整原因與差異記錄。
- 產出率、報廢、重工、替代料都屬成本與庫存風險點，不能只在備註欄處理。

### 7.3 庫存與追溯規則

- 庫存管理以批次與台帳為基礎，任何入庫、領料、完工、調整、轉倉、報廢都要有來源單據。
- 原則上不允許負庫存；若因系統切換或例外營運需要暫時放寬，必須明確定義允許角色、期限與補正機制。
- 涉及效期、批次與追溯的原料或成品，必須可追蹤到收貨或生產來源。

### 7.4 訂單、出貨與對帳規則

- 訂單、出貨、發票、收付款與核銷必須是不同但可追溯的業務節點，不可用單一狀態欄位混合表示。
- 發票與收款不可早於可被核對的來源單據；若允許預收或預開，必須有明確沖銷與補正規則。
- 對帳完成屬高風險不可逆操作，必須有操作人、時間、依據與必要的人工覆核紀錄。

### 7.5 RBAC 與職責分離

- 門市營運、採購、倉管、生產、財務與系統管理角色必須可區分，不得預設單一超級角色長期承擔全部操作。
- 影響金額、庫存、配方、權限與關帳結果的操作，至少要支援 maker-checker 或等價覆核能力。
- 稽核與營運分析角色可以讀取彙總資料，但不應直接擁有回寫交易資料的權限。
