# picking-order-analyzer 納入 Ivyhouse OP System 整合方案草稿

更新日期：2026-03-28

Authoritative source：否（draft）

## 目的

本文件用來分析是否應把 `picking-order-analyzer` 納入 Ivyhouse OP System，以及若採納，應以什麼方式整合、哪些能力可直接承接、哪些能力必須重構後才能進正式系統。

本文件是整合策略草稿，不等同正式 ADR，也不取代後續的 Phase 1 MVP scope、核心流程 spec、state machine definitions 或實作 plan。

## 結論摘要

目前建議採納的方向不是把 `picking-order-analyzer` 當成獨立工具長期並存，也不是把該 repo 原樣搬進 Ivyhouse OP System，而是把它已經被驗證有價值的核心能力，重構納入 Ivyhouse OP System 的正式模組中，作為 Phase 1 的 `batch-first` 訂單 intake / demand aggregation 能力。

這個方向可明顯降低 Phase 1 在以下面向的探索成本：

1. 多平台撿貨單格式辨識與解析
2. 商品名稱 / 規格對正式商品的映射規則
3. 每日批次需求彙總
4. 批次操作 UI 與人工映射確認流程
5. 現場既有作業方式的導入阻力

但此方向不能被理解成「直接複製現有工具即可完成正式系統需求」。若要納入 Ivyhouse OP System，仍必須補上正式主資料、審計、權限、批次追溯、來源治理與後續 handoff 能力。

## 問題背景

目前公司的訂單來源分成以下幾類：

1. 電商平台：蝦皮、MO店+、官網、生活市集、酷澎、橘點子
2. 人工來源：電話、Line 私訊、FB 粉絲團私訊
3. 現場購買

目前每日早上的作業模式不是直接從各平台 API 同步每筆訂單，而是：

1. 從鼎新 A1 或各平台下載撿貨單
2. 使用 `picking-order-analyzer` 解析蝦皮、MO店+、官網、橘點子的撿貨單
3. 將解析後的商品數量統計結果匯入自有訂單統計表
4. 將生活市集、酷澎、電話、Line、FB、現場購買等來源手動補入統計表

因此，現況真正被驗證的需求不是「即時接單系統」，而是「每日批次需求 intake 與商品數量彙總」。

## 當前前提與限制

以下前提在目前階段視為成立，後續若改變，必須同步更新本草稿與後續 scope 文件：

1. 短期內無法依賴平台 API key 取得每個平台的每一筆訂單明細。
2. 在 API key 尚未取得前，Phase 1 不應強制要求平台訂單級明細進系統。
3. 在 API key 尚未取得前，只要能透過匯入撿貨單計算商品數量，並區分來源平台，即可支撐目前主要 batch-first 作業。
4. `picking-order-analyzer` 現階段主要可自動處理四個來源：蝦皮、MO店+、官網、橘點子。
5. 生活市集、酷澎目前雖仍手動輸入，但屬可能進一步結構化的來源，不應視為永久純人工來源。
6. 現場購買同時存在「現貨直接銷售」與「預訂後進生產 / 保留庫存」兩種情境，後續不得混為同一流程。

## 對產品目標的影響

Ivyhouse OP System 的最終目標仍是「從接單到生產、包裝、出貨、對帳，全流程可追蹤」。但在 API key 尚未到位的前提下，Phase 1 可接受的暫時收斂方式應為：

1. 正式追溯起點先從「來源批次匯入」開始，而不是從平台原始訂單事件開始。
2. 系統先追蹤「來源平台」與「批次需求」，不假裝自己已擁有完整平台訂單級明細。
3. 未來 API key 到位後，再沿用同一套正式模型把追溯起點往前擴充到平台訂單層。

換句話說，這是受限條件下的暫時性 Phase 1 切法，不是最終產品邊界。

## 整合判定

### 建議採納

- 採納把 `picking-order-analyzer` 的核心能力納入 Ivyhouse OP System。
- 採納 `batch-first` 作為 Phase 1 的現實切入點。
- 採納「先支援平台來源層級與商品數量彙總，保留未來 API 直連擴充點」的策略。

### 明確不建議

- 不建議把 `picking-order-analyzer` 當成長期獨立平行系統。
- 不建議把該 repo 的前端頁面、全域狀態、檔案處理流程原樣搬進正式系統。
- 不建議在沒有 API key 的前提下，虛構或強補平台訂單級明細欄位，製造假的完整性。
- 不建議讓日統計表直接成為唯一正式資料來源，而沒有原始來源檔、匯入批次與映射結果的追溯鏈。

## 建議納入的正式能力名稱

建議把這一塊能力在 Ivyhouse OP System 中正式命名為：

`Batch Intake / Demand Aggregation`

它的責任不是完整取代未來的 platform order sync，而是承接目前真實存在的 daily batch 作業，將多來源撿貨單與手動補單結果，轉成可被生產、庫存、包裝與出貨流程正式引用的需求輸入。

## 建議承接的功能範圍

### Phase 1 建議直接承接

1. 多平台撿貨單檔案上傳
2. 平台檔案格式識別
3. 撿貨單解析
4. 商品名稱 / 規格映射到正式銷售商品
5. 人工覆核與修正映射
6. 批次需求彙總
7. 匯出批次統計結果與作業報表
8. 記錄來源平台、匯入時間、操作者、來源檔案與結果版本

### Phase 1 應先保留但不強做

1. 平台訂單 API 直連
2. 平台訂單級明細同步
3. 客戶資料與每筆平台訂單的完整履約追溯
4. 平台狀態回寫
5. 平台取消 / 退款 / 拆單 / 併單的即時同步

### Phase 1 需補充但不可缺省

1. 匯入批次審計
2. 原始來源檔保存或可追蹤引用
3. 映射規則版本化
4. 手動補單來源標記
5. 匯入失敗、重跑與覆核機制
6. 匯入結果到後續生產 / 庫存流程的正式 handoff

## 模組歸屬建議

| 模組 | 建議責任 |
|------|------|
| Portal / Identity / Audit | 上傳者、覆核者、操作紀錄、批次審計 |
| Master Data | 銷售商品主檔、平台商品映射規則、渠道主資料 |
| Order / Fulfillment | 來源批次 intake、需求彙總、手動補單、批次輸出與後續履約 handoff |
| Production | 承接已確認的批次需求，轉排程 / 工單 |
| Inventory | 根據後續工單、包裝、出貨承接正式庫存異動 |
| Analytics / Reporting | 查詢批次彙總結果與例外，不得反向改寫正式輸入 |

此能力不建議被切成獨立平行模組；它應以 `Order / Fulfillment` 為主責，並依賴 `Master Data` 與 `Portal / Identity / Audit`。

## 建議新增的正式實體

以下實體雖未必都在 Phase 1 一次做滿，但若要把既有工具正式整進系統，至少要有對應資料模型或權威定義：

1. `銷售渠道`：平台 / 電話 / Line / FB / 現場等來源分類
2. `來源平台類型`：蝦皮、MO店+、官網、橘點子、生活市集、酷澎等
3. `來源檔案`：被上傳的撿貨單或其他來源文件
4. `匯入批次`：一次批次 intake 的主紀錄
5. `匯入批次明細`：每個來源檔或手動來源在該批次中的子紀錄
6. `解析結果`：從來源檔解析出的原始商品行
7. `映射結果`：原始商品行對正式商品 SKU 的映射紀錄與信心 / 覆核結果
8. `批次需求彙總`：可正式交給後續流程的商品需求數量
9. `手動補單紀錄`：對生活市集、酷澎、電話、Line、FB、現場等非自動來源的結構化輸入

## 關鍵設計原則

### 1. 不假裝擁有不存在的訂單級明細

在 API key 尚未取得前，系統應誠實承認目前掌握的是「來源平台層級的批次需求」，而不是完整的原始訂單級資料。

### 2. 平台來源要保留，但不必強做訂單級主鍵

現階段至少要能回答：

1. 這批需求來自哪個平台
2. 哪個來源檔或哪筆手動補單
3. 由誰在何時匯入
4. 映射後形成哪些正式商品需求

### 3. 映射規則必須正式化

`picking-order-analyzer` 目前已累積的商品名稱 / 規格映射邏輯，不應只留在前端規則檔；納入正式系統後，必須成為可審計、可版本化、可覆核的正式規則資產。

### 4. 日統計結果不是唯一真相

最終正式引用給後續流程的可以是批次需求彙總，但系統仍必須保留其來源鏈：來源檔、解析結果、映射結果、人工修正紀錄。

### 5. 未來 API 模式必須是擴充，不是推翻

本次整合若設計得當，未來平台 API key 到位後，應是在同一套 `銷售渠道 -> 來源事件 / 匯入 -> 正式需求` 模型上往前延伸，而不是重做另一套 intake 系統。

## 對 `picking-order-analyzer` 的承接方式建議

### 可直接吸收的資產

1. 平台 parser 行為與格式知識
2. 商品映射規則經驗
3. 批次操作流程
4. 報表輸出需求
5. 例外處理與人工修正經驗

### 應重構後吸收的資產

1. 檔案上傳與前端流程
2. 全域狀態管理
3. 匯出報表資料流
4. 平台識別與解析器介面
5. 商品映射規則儲存方式

### 不應直接沿用的部分

1. 將工具視為最終正式資料庫的替代品
2. 只有前端記憶體狀態、沒有正式持久化與審計的處理方式
3. 缺 RBAC、缺 maker-checker、缺批次治理的操作模型

## 建議的 Phase 1 收斂版本

若以現況最務實的方式收斂，建議 Phase 1 的這塊能力先以以下版本落地：

1. 支援蝦皮、MO店+、官網、橘點子撿貨單匯入
2. 支援生活市集、酷澎的半結構化手動匯入
3. 支援電話、Line、FB、現場來源的人工補單
4. 將所有來源統一轉成「平台 / 來源可辨識」的批次需求
5. 批次需求經人工確認後，可交給後續生產 / 備貨流程使用
6. 暫不要求平台原始訂單級明細與客戶級履約追溯

這代表 Phase 1 在 intake 端的正式目標應是：

`可追溯的批次需求輸入`，而不是 `完整平台訂單中台`

## 主要風險

1. 若只把現有工具搬進來而不補正式資料模型，後續仍無法支撐生產、庫存與審計整合。
2. 若過早要求平台訂單級明細，會把 Phase 1 依賴綁死在外部 API key 取得進度上。
3. 若不保留來源批次與映射版本，日後會無法解釋需求數量從何而來。
4. 若手動來源沒有結構化欄位，系統最終仍會退回自由文字與 Excel 外掛流程。
5. 若現場購買不拆成現貨銷售與預訂兩類，後續狀態模型會持續混亂。

## 建議後續文件化順序

1. 在正式 Phase 1 MVP scope 中納入 `Batch Intake / Demand Aggregation` 能力
2. 補一份 intake / demand aggregation flow spec
3. 補 `銷售渠道`、`匯入批次`、`來源檔案`、`批次需求彙總` 的主資料 / 交易實體定義
4. 補映射規則治理與批次覆核的 RBAC / approval 邊界
5. 再進入實際 app bootstrap 與實作拆解

## 本草稿希望回答的核心問題

### 可以把 `picking-order-analyzer` 做進 Ivyhouse OP System 嗎？

可以，但應以「重構式整合」納入正式模組，不應原樣移植或長期平行維運。

### 這樣能否省掉 batch-first 的大半工作？

可以省掉大半的格式探索、解析規則與操作流程試錯，但省不掉正式資料模型、審計、權限、批次治理與後續模組 handoff 的工作。

### 在 API key 尚未取得前，是否可以只分平台來源與商品數量？

可以。現階段應把這視為明確的 Phase 1 暫時前提，並在模型上保留未來往平台訂單級明細擴充的空間。
