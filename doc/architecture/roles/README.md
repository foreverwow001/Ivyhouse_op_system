# RBAC 與 Approval Matrix

更新日期：2026-04-03

Authoritative source：是

## 目的

本文件是 Ivyhouse OP System 在 Phase 0 的第一版 RBAC 與 approval matrix。這一版先覆蓋目前四表直連的高風險主資料 / 規則治理操作，並補上 Phase 1 所需的 Batch Intake / 需求規劃權限分層原則，明確區分：

- 日常訂單處理與補單修正的帳號存取層級
- 主資料 / 規則 owner 的正式治理權限

避免把「可以操作某一批訂單」誤解成「可以修改正式主資料或永久規則」。

## 範圍

- 本版只涵蓋四表及其直接相關高風險操作：
  - 外包裝材料主檔
  - 內包裝完成品主檔
  - 單位換算規則表
  - 生產 / 分裝 / 轉換扣帳規則表
- 本版不處理登入、API、頁面、完整資料列層級授權，也不替代全系統 RBAC 終版。
- 本版不擴張到採購、工單、庫存調整、財務對帳等全域流程，但會保留與其相容的 approval 邊界語意。
- 本版新增 Phase 1 的權限分層補充，涵蓋：Batch Intake、補單 / 修正批次、單筆映射覆核、旺季試算工作區，以及銷售渠道 / 商品映射規則 / 渠道時效規則的治理邊界。

本版只採用目前權威文件已明示的角色：

| `採購` | 供應、物料與採購流程 | 外包裝材料主檔的提案 / 維護發起角色 |

| `生產` | 生產排程、工單、領料、完工、報廢 | 內包裝主檔與規則內容的主要提案 / 維護發起角色 |

## 帳號存取層級（Phase 1 日常作業）

Phase 1 的日常訂單處理，另外存在一條「帳號存取層級」軸線。這條軸線處理的是誰可以操作 Batch Intake、補單 / 修正批次與旺季試算，不直接決定誰可以修改正式主資料。

| 帳號層級 | 主要用途 | 說明 |
|----------|----------|------|
| `行政` | 日常訂單處理 | 可進行上傳、手動輸入、補單、修正批次、單筆映射覆核、試算操作 |
| `會計` | 日常訂單處理與查核 | 與 `行政` 同樣可操作日常訂單處理；另承接對帳前的需求確認查核 |
| `主管` | 日常訂單處理與管理檢視 | 與 `行政`、`會計` 同樣可操作日常訂單處理，並可查看試算與正式需求結果 |

說明：

- 同一位使用者可以同時擁有一個帳號存取層級，以及一個或多個業務責任角色。
- 帳號存取層級只處理日常作業存取，不自動授予正式主資料治理權限。

### 現況與治理規則的落差

- 以上矩陣描述的是目前已確認的營運現況，不代表已滿足終版 RBAC 與 maker-checker 要求。
- 依 `project_rules.md`，影響庫存、配方、權限與關帳結果的操作，長期仍應具備 maker-checker 或等價控制能力。
- 因此，目前這批 `一般操作、無覆核` 的規則，應視為 Phase 0 / Phase 1 現況基線，而不是終版治理完成狀態。
- 後續若進入採購、財務、正式庫存調整與結帳模組實作，需再把這一層現況 SOP 收斂到更正式的 approval boundary。

- 在完整 auth framework 落地前，inventory-count 的高風險 API 可先使用 request header `x-ivyhouse-role` / `x-ivyhouse-roles` 實作最小權限閘。

### Phase 1 production-planning 過渡 guard

- 在完整 auth framework 落地前，production-planning 的高風險 API 可先使用 request header `x-ivyhouse-role` / `x-ivyhouse-roles` 實作最小權限閘。
- production-planning 本修正輪的正式角色名稱為：`生產`、`包裝及出貨`、`會計`、`管理員`。

- 這個 guard 只屬 Phase 1 過渡控制，不等於正式登入、身份鑑別與 permission 終版。

## Idx-021 正式身份與 approval skeleton 決策

以下決策用於 `Idx-021` 第一輪正式 auth / maker-checker 補強，約束 inventory-count 與 production-planning 的正式身份承接與 approval skeleton。

### 正式身份來源

- 第一輪正式身份來源採內部 Portal session principal。
- principal 至少應保留：`principalId`、`displayName`、`roleCodes`、`sessionId`、`authSource`。
- Phase 1 過渡 guard 使用的 `x-ivyhouse-role` / `x-ivyhouse-roles` 只作過渡控制，不得再視為正式身份來源。
- 後續若需要背景作業或排程直接寫入 daily-ops，可在同一 principal contract 下擴充 service principal，但不應反向影響人員操作的第一信任來源。

### `主管` 與 `管理員` 分工

- `主管` 是正式業務角色，負責高風險日常營運核准。
- `管理員` 是系統治理角色，負責權限設定、系統維運與緊急處置，不得作為高風險業務最終 approver。
- 同一個使用者帳號可同時持有 `主管` 與 `管理員`；但系統在判斷高風險業務核准時，必須依 `主管` 角色成立，不得因同時具備 `管理員` 身分而繞過。

### `admin + supervisor` 同帳號交集政策

- 當同一帳號同時具有 `管理員` 與 `主管` 時，高風險業務 approval 一律以 `主管` 身分認定。
- `管理員` 身分不得放寬高風險業務 approver 邊界，也不得作為 maker-checker 的額外放行條件。
- 系統記錄 approver 時，應保留 approval 當下的完整 `roleCodes`；但核決是否合法，只能看該動作要求的正式業務角色是否成立。
- Phase 1 的單人營運例外只允許依 `主管` 身分成立的單人 override；若同一帳號同時帶有 `管理員`，不得把 `管理員` 視為 override 合法性的來源。
- 涉及系統維運、權限設定、緊急處置等非業務核准動作時，仍可由 `管理員` 身分執行；但這類行為不得混入 production-planning、inventory-count 等高風險業務 approval audit。

### 第一輪高風險業務核准範圍

- 第一輪先納入 `主管` 邊界的高風險動作：
  1. `inventory-count complete session`
  2. `inventory-count manual adjustment`
- `production-planning` 的 create / revise / rerun 先維持 Phase 1 過渡 guard，是否進入正式 `主管` approval scope，待下一輪與 `Idx-021` 後續切片再定。

### 單人營運例外

- 單人營運期間，允許 maker 與 checker 為同一人。
- 這種情況不得無痕通過；至少必須留下 `single_person_override` 或等價 audit 標記。
- 若同一帳號同時具 `主管 + 管理員`，單人 override 仍只以 `主管` 邊界成立，不得以 `管理員` 身分補足 approver 資格。
- 因此，Phase 1 第一輪的目標不是強制雙人覆核，而是先把角色模型、actor / maker / checker 欄位責任與例外稽核線索建立起來。

## 權限語意

| 縮寫 | 說明 |
|------|------|
| `V` | 可查閱 |
| `O` | 可操作日常作業，但不代表可維護 owner 主資料 |

| `A` | 可作最終核定 |
| `X` | 不可執行 |

## 雙層權限模型

Phase 1 開始，需明確區分兩層權限：

1. 帳號存取層級：決定誰可以操作日常 Batch Intake、補單 / 修正批次、單筆映射覆核與旺季試算。
2. 業務責任角色：決定誰可以新增、修改、停用正式主資料與高風險規則，並承擔 maker-checker 責任。

正式主資料與永久規則的發布、停用與核定，永遠以 owner module 與 approval matrix 為準，不以帳號層級直接取代。

## Approval Matrix

### 核心原則

- maker 與 checker 不得由同一角色承接。
- 同一工作單位中的同一筆高風險變更，不得由同一人先以 maker 身分提出、再以 checker 身分核定；若需換手，必須留下重新指派與交接證據。
- `核定人` 欄位記錄最終核定角色名，不記錄 maker。
- `系統管理` 可配置權限，但不得作為任何高風險業務資料或規則的最終核定角色。
- 若變更會影響既有銷售商品引用、庫存扣帳或正式規則，必須先完成相鄰 consumer 影響盤點，再進入核定。

Idx-021 補充：

- 對 inventory-count 與 production-planning 的正式 auth / approval skeleton 而言，原則上仍要求 maker 與 checker 分離。
- 但在單人營運期間，若同一 principal 同時承擔 maker 與 checker，系統必須明確留下 `single_person_override` 或等價 audit 標記，而不是假裝符合雙人覆核。
- `管理員` 不得因系統維運權限而成為 inventory-count 或 production-planning 的高風險業務最終 approver。

| 高風險操作 | Maker 角色 | Checker / Approver 角色 | 直接影響 | `核定人` 欄位正式值 |
|------------|------------|-------------------------|----------|----------------------|
| 外包裝材料主檔新增或大幅修改 | `採購` | `生產` | 銷售商品組成、外包裝引用 | N/A |
| 外包裝材料主檔停用 | `採購` | `生產` | 現行包裝 consumer 切換 | N/A |
| 內包裝完成品主檔新增或大幅修改 | `生產` | `倉管` | 換算、扣帳、銷售商品組成 | N/A |
| 內包裝完成品主檔停用 | `生產` | `倉管` | 現行組成與規則引用切換 | N/A |
| 單位換算規則新增 / 修改 / 停用 | `生產` | `倉管` | 庫存數量語意、扣帳、成本基礎 | `倉管` |
| 轉換扣帳規則新增 / 修改 / 停用 | `生產` | `倉管` | 生產扣帳、重量中介、轉換鏈 | `倉管` |

Phase 1 補充：`銷售渠道`、`商品映射規則`、`渠道時效 / cutoff 規則` 已先在本文件明定 owner module 與日常處理邊界，但其 maker-checker 指派仍待 RBAC v2 與實際 Order / Fulfillment owner 人員任命後補正式指定。在此之前，這三類資料不得因帳號層級較高而繞過 owner module 治理直接發布正式值。

## `核定人` 驗證規則

- 規則表允許的 `核定人` 正式值，只能來自本文件角色清單中的正式角色名。
- Phase 0 第一版目前允許落地到規則表的正式核定角色，僅有 `倉管`。
- 若 `核定人` 不存在於本文件，或與 approval matrix 不一致，該列不得視為 `正式核定`。
- `核定人` 值的文件驗證責任由 Master Data owner 與規則 owner 共同承擔；後續若進入系統化驗證，應由 owner service 在寫入時阻擋無效角色值。

## 查閱範圍補充

- `門市營運`：僅需查閱 SKU、名稱、狀態與用途，不預設可見核定依據、損耗率或其他高風險規則細節。
- `財務`：以查閱規則結果與追溯依據為主，可用於成本與對帳查核，但不直接回寫規則內容。
- `稽核與營運分析`：可查閱完整欄位，以支援例外追蹤與審計。

## 四表回收規則

## 限制與待補事項

- 本版是角色層級，不是人員層級；後續若有正式人員任命或部門對應，再往下擴充。
- 本版未處理列層級資料範圍、登入授權、API 保護或 UI 可見性。
- 付款、核銷、對帳差異、庫存調整、配方發布等全域高風險操作，仍需在後續版本補齊。
- 本版已補上 Batch Intake 與旺季試算的帳號存取層級，但尚未進入人員層級任命、部門映射與系統實際授權設定。
- 本版已補上日常營運四角色的現況操作矩陣，但此矩陣仍需在後續版本與 maker-checker 治理要求進一步收斂。
