## 對整合草稿的分析與修正

在起草本 MVP scope 前，先記錄整合草稿經後續業務確認後需要修正的事項：

### 1. 鼎新 A1 的定位修正

整合草稿中寫「從鼎新 A1 或各平台下載撿貨單」，語感暗示 A1 是某種 ERP 或系統。

經確認：鼎新 A1 只是部分平台（主要是蝦皮）的撿貨單下載工具，不是 ERP，也沒有正式資料庫、權限、審計或流程治理能力。Ivyhouse OP System 是公司第一套正式 ERP 級系統。

影響：Phase 1 的定位不是「承接既有 ERP 的部分功能」，而是「建立公司第一套正式營運資料骨幹」。

### 2. Excel 統計表的功能遠超「日統計」

整合草稿將「訂單統計表」定義為日商品總量的統計結果。

經確認：這張 Excel 統計表實際上是一張滾動式的需求 / 庫存 / 生產決策工作表，功能包括：

1. 每週五實體盤點一次，取得基線庫存
2. 週一把盤點數字填入統計表，作為本週起始庫存
3. 每天扣除當天訂單需求量，得到扣除需求後的庫存餘量
4. 週二複製週一扣除後庫存到週二的統計表，以此類推到週五
5. 每個商品有設定最低庫存量（安全庫存）
6. 人工判斷今天要生產什麼、做多少

因此，新系統必須取代的不只是 `picking-order-analyzer`，還有這張統計表所承載的「需求匯入 + 庫存扣除 + 生產決策參考」整體作業。

### 3. 批次頻率修正

整合草稿以「每日早上一批」為假設。

經確認：一天可能多批。模型必須支援同一天多次匯入。

### 4. 完全取代，無平行過渡期

整合草稿未明確承諾取代時程。

經確認：Phase 1 上線後，`picking-order-analyzer` 直接停用，不做平行運作。這代表 Phase 1 在 intake 端的完成度必須至少等同現有工具，且提供額外的持久化、審計與追溯能力。

### 5. 缺少「需求扣庫存」核心能力

整合草稿的 Phase 1 能力範圍只到「批次需求彙總」，沒有涵蓋「需求扣庫存 → 建議生產量 → 人工決策」這段。

經確認：這段是目前每天最核心的作業決策依據。Phase 1 若不覆蓋這段，統計表無法被取代。

### 6. 缺少安全庫存概念

整合草稿未提到安全庫存 / 最低庫存量。

經確認：目前每個商品都有在 Excel 中設定最低庫存量。Phase 1 必須支援此概念。

### 7. 生產排程有當天做與隔天做兩種模式

整合草稿未提及。

經確認：部分商品當天統計完當天生產，部分商品隔天或後天才生產。Phase 1 的需求到生產 handoff 必須支援這兩種模式。

### 8. 現場購買有既有紀錄

整合草稿假設現場購買紀錄較不結構化。

經確認：有 Excel 或手寫單紀錄形式。Phase 1 應納入現場購買的結構化記錄能力。

### 9. 銷售渠道 / 來源平台類型 語意簡化建議

整合草稿提出 `銷售渠道` 與 `來源平台類型` 為兩個獨立實體。

建議合併為單一 `銷售渠道` 主資料，以類型欄位區分電商平台 / 人工來源 / 現場渠道，避免過度拆分。

### 10. 手動補單不必獨立實體

整合草稿提出 `手動補單紀錄` 為獨立實體。

建議將手動補單視為 `匯入批次明細` 的一種 method = manual 子型態，不需要獨立實體。

## Excel 生產統計表分析結論與範圍影響

本節根據實際營運 Excel（`20260325-生產統計表.xlsm`，3 個 tab、535 條公式）的分析結果，記錄對 Phase 1 scope 的範圍級影響。詳細公式對照應另建 implementation spec，不在本文件範圍。

### Excel 三層結構

| Tab | 名稱 | 實際功能 | 公式數 |
|-----|------|----------|--------|
| 1 | 出貨統計表 | 需求匯入 + BOM 拆分（禮盒 → 組件品） | 178 |
| 2 | 餅乾庫存(製作) | 庫存快照 + 包裝量 + 急單量追蹤 | 120 |
| 3 | 餅乾庫存(叫貨) | 需求扣庫存 + 安全庫存 + 倍數制生產建議 | 237 |

三層之間以公式鏈互相引用：Tab1 彙總需求 → Tab2 庫存加減 → Tab3 生產決策。

### 範圍級發現與處理

| # | 發現 | 對 Phase 1 的影響 | 處理方式 |
|---|------|-------------------|----------|
| E1 | **BOM 拆分**：禮盒需求透過公式自動拆分為個別品項（例：綜花餅乾禮盒 → 各口味餅乾片數） | Phase 1 必須支援自動 BOM 拆分，否則無法取代 Excel Tab1 的核心功能 | 納入能力 A5、B2，引用銷售商品組成對照表 |
| E2 | **倍數制生產**：每個自製品項有「單倍產量」基數，生產建議以「倍」為單位（例：曲奇 1 倍 = 特定公斤數） | Phase 1 必須支援，此為生產決策的基本語言，也是未來排程模組的基礎 | 納入能力 B3 + 主資料（內包裝完成品新增「單倍產量」欄位） |
| E3 | **急單（即期量）**：「即期」= 當天必出貨的急單數量，不是商品保存期限 | Phase 1 需在需求確認時支援急單標記，生產建議中急單品項優先排序 | 納入能力 A5、B2、B3 |
| E4 | **自製 vs 委外**：Excel 中 自製品走「製作」欄，委外品走「進貨量」欄，流程不同 | Phase 1 只做自製品項；委外品排除 | 新增前提 P9、不做清單 N14 |
| E5 | **多單位追蹤**：同品項以 4 種單位形式追蹤（真空袋 kg → 片 → 90g → 135g） | 既有單位換算規則表已涵蓋，但需求扣庫存與生產建議的呈現必須使用該品項的實際作業單位 | 納入能力 B2 的單位呈現要求 |
| E6 | **負庫存已存在**：Excel 中已出現負庫存（例：椰棗核桃 = -3） | 系統設計不能假設庫存永遠非負；Phase 1 應允許顯示但標記警告，不硬擋 | 納入能力 B1 補充說明 |
| E7 | **跨日滾動靠人工複製貼上**：Tab2/Tab3 的庫存每天由前一天手動搬到當天欄位 | 系統自動滾動庫存已在能力 B1 涵蓋，此發現驗證了需求的必要性 | 已涵蓋 |
| E8 | **Tab 名稱與實際功能不符**：「出貨統計表」其實是需求匯入 + BOM 拆分；「叫貨」其實是生產決策 | 文件和系統命名不應沿用 Excel 的誤導名稱 | 命名原則備忘 |

### 不納入 Phase 1 scope 但需產出的 Phase 0 前置文件

| 文件 | 說明 |
|------|------|
| Excel → 系統公式對照表 | 將 Tab1-Tab3 的關鍵公式鏈對應到系統計算邏輯，作為 implementation spec 的一部分 |
| 倍數制基數資料 | 從 Excel 提取每個自製品項的單倍產量，轉為主資料遷移種子資料 |
| 安全庫存資料 | 從 Excel Tab3 提取每品項安全庫存量，轉為主資料遷移種子資料 |

## Phase 1 定位

### 這不是什麼

- 不是完整的平台訂單中台。
- 不是財務深度整合系統。
- 不是完整供應鏈最佳化工具。
- 不是承接既有 ERP 的局部對接。

### 這是什麼

公司第一套正式 ERP 級營運系統的第一個可工作版本，目標是：

> 用 batch-first 方式把每日多來源需求匯入、庫存扣除、生產決策、基本包裝出貨串成一條可追溯的主線，取代目前的 picking-order-analyzer + Excel 統計表 + 人工判斷的分散作業。

## 當前前提與約束

以下前提在 Phase 1 期間視為成立。若任何一項發生變化，必須重新評估 Phase 1 scope：

| # | 前提 | 影響 |
|---|------|------|
| P1 | 平台 API key 短期內無法取得 | 不能依賴平台訂單級明細；只能從撿貨單解析平台來源與商品數量 |
| P2 | 鼎新 A1 只是撿貨單下載工具，不是 ERP | 新系統是第一套正式 ERP，不是對接既有 ERP |
| P3 | Phase 1 上線後完全取代 picking-order-analyzer，不做平行運作 | intake 功能完成度必須至少等於現有工具 |
| P4 | 庫存基線來自每週五實體盤點 | 系統庫存精度受盤點週期限制；Phase 1 不追求即時精準庫存 |
| P5 | 生產決策最終由人工判斷 | 系統提供需求扣庫存的建議數量，但不自動排程 |
| P6 | 一天可能多批匯入 | 批次模型不可假設每日只匯入一次 |
| P7 | 現場購買包含「現貨直售」和「預訂排產」兩種模式 | 不可混為同一流程 |
| P8 | 生產排程有「當天做」和「隔天做」兩種模式 | 需求到生產 handoff 的時間間隔不固定 |
| P9 | Phase 1 只處理自製品項，委外品項留待 Phase 2 | 委外品項（如椰棗核桃、牛軋糖）在 Phase 1 不產生工單，不進入生產建議 |
| P10 | 旺季試算與正式營運主線邏輯分離 | 試算不改正式庫存、不產生正式工單；兩者共用主資料但生命週期各自獨立 |

## Phase 1 MVP 目標

Phase 1 MVP 的正式目標是打通以下主線：

```
多來源需求匯入 → 商品映射 → BOM 拆分 → 批次需求彙總（含急單標記）
→ 需求扣庫存 → 建議生產量（倍數制，急單優先）
→ 人工確認排程 → 生產工單 → 扣料 → 完工入庫
→ 包裝 → 出貨 → 批次需求狀態回寫
```

這條主線與 `project_overview.md` 核心營運主線的差異在於：正式追溯起點從「來源批次匯入」開始，而不是從「平台原始訂單」開始。這是 P1 前提下的暫時收斂，不是最終產品邊界。

## Phase 1 核心能力

### 能力 A：Batch Intake / Demand Aggregation

`Owner: Order / Fulfillment`

這是 Phase 1 的核心入口能力，承接 `picking_order_analyzer_integration_draft.md` 的整合結論。

#### A1. 撿貨單匯入與解析

| 項目 | 說明 |
|------|------|
| 支援平台 | 蝦皮（PDF）、MO店+（Excel/PDF）、官網（Excel）、橘點子（Excel） |
| 功能 | 上傳撿貨單檔案 → 自動識別平台 → 解析商品名稱與數量 |
| 來源 | 重構 `picking-order-analyzer` 的 parser 與 ParserFactory |
| 驗收 | 至少支援現有工具已支援的四個平台格式，解析結果正確率不低於現有工具 |

#### A2. 半結構化手動匯入

| 項目 | 說明 |
|------|------|
| 支援平台 | 生活市集、酷澎 |
| 功能 | 上傳來源報表或手動輸入結構化表格（來源平台、商品名稱、數量） |
| 驗收 | 可匯入並標記來源平台，結果納入同一批次需求 |

#### A3. 純手動補單

| 項目 | 說明 |
|------|------|
| 支援來源 | 電話、Line、FB、現場購買 |
| 功能 | 手動輸入商品名稱（或選擇正式商品）、數量、來源渠道 |
| 現場購買 | 必須區分「現貨直售」與「預訂排產」兩種類型 |
| 驗收 | 所有手動來源納入同一批次體系，可追溯操作者與時間 |

#### A4. 商品映射

| 項目 | 說明 |
|------|------|
| 功能 | 自動將解析出的商品名稱 / 規格映射到正式銷售商品 SKU |
| 人工覆核 | 映射信心不足或新商品時，必須有人工確認介面。此處確認的是「本批次映射結果」，不是直接修改主資料規則 |
| 規則治理 | 映射規則必須可維護、可版本化、有啟用 / 停用狀態；永久規則維護屬 Master Data owner 權責，與批次處理權限分離 |
| 來源 | 重構 `picking-order-analyzer` 的 MappingEngine 與各平台映射規則 |
| 驗收 | 已知商品的映射正確率不低於現有工具 |

#### A5. 批次需求彙總與 BOM 拆分

| 項目 | 說明 |
|------|------|
| 彙總 | 將同一批次中所有來源的映射結果，彙總為「商品 × 數量」的需求清單 |
| BOM 拆分 | 禮盒類銷售商品必須根據「銷售商品組成對照表」自動拆分為內包裝完成品層級的組件需求（例：綜花餅乾禮盒 → 各口味餅乾 × 對應片數）。拆分後的組件需求與原始禮盒需求同時保留，便於追溯 |
| 急單標記 | 當天必出貨的需求（即期量）可在確認時標記為急單，後續流程可識別急單優先權 |
| 支援同一天多批 | 每批獨立可辨識，可合併查看當日所有批次的累計需求 |
| 人工確認 | 彙總結果須經操作者確認後，才正式成為後續流程的輸入 |
| 驗收 | 可追溯每行需求來自哪個批次、哪個來源檔或手動輸入；禮盒需求可追溯到拆分後的組件明細 |

#### A6. 來源追溯鏈

| 項目 | 說明 |
|------|------|
| 必須保留 | 來源檔案（或儲存副本）、解析結果、映射結果、人工修正紀錄 |
| 追溯問題 | 任何一行批次需求，必須能回答：來自哪個平台、哪個檔案、解析出哪些原始行、映射到哪個正式商品、由誰在何時確認 |
| 驗收 | 可從最終批次需求反查來源鏈的完整步驟 |

### 能力 B：需求扣庫存與生產建議

`Owner: Inventory + Production`

這是 Phase 1 取代 Excel 統計表的核心能力，是 Batch Intake 到 Production 之間的橋樑。

#### B1. 庫存基線管理

| 項目 | 說明 |
|------|------|
| 庫存來源 | 每週實體盤點結果作為基線；日常出庫、完工入庫、盤點調整 |
| 安全庫存 | 每個銷售商品（或內包裝完成品）可設定最低庫存量 |
| 庫存滾動 | 系統自動根據已確認的出庫、入庫、調整計算當前可用庫存，取代 Excel 的每日人工複製 |
| 負庫存處理 | 系統允許顯示負庫存但必須標記警告（現行 Excel 已存在負庫存案例），不強制阻擋 |
| 盤點 | 支援定期實體盤點結果輸入，產生盤盈盤虧記錄 |
| 驗收 | 庫存餘量可追溯到最近一次盤點基線與後續異動明細 |

#### B2. 需求扣庫存計算

| 項目 | 說明 |
|------|------|
| 功能 | 當批次需求確認後，自動計算：需求量 - 可用庫存 = 淨需求 |
| 庫存計算粒度 | 預設以 BOM 拆分後的內包裝完成品層級為主；但對直接以銷售商品層級管理庫存的品項，允許保留銷售商品層級。實際粒度由商品類型與主資料設定決定，禮盒需求仍須先在 A5 完成拆分 |
| 急單標記 | 標記為「即期」（當天必出貨）的需求，在扣庫存計算結果中優先顯示並標示急單狀態 |
| 單位呈現 | 系統須根據單位換算規則，以該品項實際生產 / 庫存使用的單位呈現（例：餅乾以「片」、綜花餅乾以「斤」、堅果類以「kg」） |
| 安全庫存警示 | 若扣除後庫存低於安全庫存，系統需標示警告 |
| 負庫存處理 | 系統允許顯示負庫存但必須標記警告（現行 Excel 已存在負庫存案例），不強制阻擋 |
| 呈現 | 提供「商品 × 當前庫存 × 累計需求 × 淨需求 × 安全庫存 × 建議生產量 × 急單量」的列表 |
| 驗收 | 計算結果至少等同目前 Excel 統計表的扣庫存邏輯 |

#### B3. 生產建議量

| 項目 | 說明 |
|------|------|
| 功能 | 系統根據淨需求與安全庫存，計算建議生產量 |
| 倍數制 | 每個自製品項有「單倍產量」基數（主資料設定）。建議生產量以「倍數」呈現（例：建議 3 倍 = 3 × 單倍產量）。人工決策時可以倍數為單位調整。此基數為未來排程模組的基礎 |
| 急單優先 | 含急單需求的品項在生產建議列表中優先排序，並標示急單量與急單來源批次，供人工優先排產決策 |

| 人工決策 | 最終生產量由人工調整確認，系統不自動排程 |
| 排程模式 | 支援「當天需求當天做」及「今天需求隔天（或後天）做」兩種模式 |
| 驗收 | 人工確認後的生產決定，可正式轉為生產工單；建議量以倍數呈現；急單品項優先排序 |

### 能力 G：旺季試算與需求規劃（支援性能力）

`Owner: Production / Planning`

此能力與正式營運主線（能力 A–F）邏輯分離，但共用同一套主資料、BOM 拆分與單位換算。試算不直接觸發正式扣庫存、不自動產生正式工單。

#### 背景

旺季（中秋節、過年）需求可達淡季數倍，客人指定出貨日密集，產能無法單日滿足。操作者需提前 1–4 週預算未來需求與庫存壓力，目前以 Excel 手工完成，旺季依賴度極高。

#### G1. 規劃期間管理

| 項目 | 說明 |
|------|------|
| 最小單位 | 日 |
| 典型範圍 | 未來 1–4 週 |
| 多期間並存 | 可同時存在多個規劃期間 |
| 狀態 | 進行中 / 已封存 |

#### G2. 試算需求匯入與異動

| 項目 | 說明 |
|------|------|
| 來源 | 所有平台數量皆可納入試算。已有指定出貨日者沿用；沒有指定出貨日的平台需求，則由操作者分配到規劃期間中的某一日 |
| 匯入能力 | 共用正式 Intake 的檔案上傳與解析基建，分流點在匯入目標選擇（試算 vs 正式） |
| BOM 拆分 | 自動執行，與正式流程一致 |
| 異動 | 追加需求、扣減需求、修改商品或數量；每次異動自動保存版本快照 |

#### G3. 試算庫存推演

| 項目 | 說明 |
|------|------|
| 起始庫存 | 取正式庫存台帳的當前可用庫存（唯讀） |
| 推演 | 預設以試算版本自身需求做逐日推演；正式已確認需求可作為唯讀參考疊加顯示，但不與試算資料自動扣抵或合併 |
| 警示 | 低於安全庫存或為負時標示 |
| 不影響正式帳 | 推演結果不改正式庫存台帳 |

#### G4. 版本歷史與分享

| 項目 | 說明 |
|------|------|
| 版本 | 每次異動自動保存，所有版本永久保留，可比對差異 |
| 分享 | 行政、會計、主管三個角色均可查看，不需正式簽核 |

#### G5. 與正式主線的邊界

| 規則 | 說明 |
|------|------|
| 不改正式庫存 | 試算不從正式庫存台帳扣減任何數量 |
| 不產生正式工單 | 試算不能直接轉為工單 |
| 不取代人工決策 | 每日實際生產量仍由人工決定，試算只提供參考 |
| 不自動對沖正式需求 | 同商品同日期在試算與正式可並存；系統不自動視為同一筆需求 |

#### 驗收基線

旺季試算列為 Phase 1 支援性能力，獨立驗收（S1–S14），不納入正式主線驗收（V1–V16）。完整驗收項目見 `demand_planning_spec.md`。

## 對 `project_overview.md` 核心營運主線的修正建議

目前 `project_overview.md` 的核心營運主線為：

```
訂單 → 生產排程 → 原料扣料/領料 → 生產完成 → 包裝 → 出貨 → 發票/收付款/核銷/對帳
```

Phase 1 的實際主線應修正為：

```
多來源需求匯入 → 商品映射 → BOM 拆分 → 批次需求彙總（含急單標記）
→ 需求扣庫存 → 建議生產量（倍數制，急單優先）
→ 人工確認 → 工單 → 領料/扣料 → 完工入庫
→ 包裝 → 出貨 → 批次需求狀態回寫
```

差異：

1. 起點從「訂單」改為「多來源需求匯入」，因 Phase 1 尚無平台訂單級明細
2. 新增「BOM 拆分 → 批次需求彙總 → 需求扣庫存 → 生產建議」段，這是 Excel 統計表的正式替代
3. 生產建議以倍數制呈現，急單優先排序
4. 終點從「發票 / 對帳」改為「批次需求狀態回寫」，因 Phase 1 不做財務

未來 API key 到位後，主線起點應往前延伸為：平台訂單 → 訂單確認 → ...，在同一套模型上擴充，不需推翻 Phase 1 架構。

```
0. 需求 Intake（撿貨單匯入 / 手動補單 / 現場購買）
1. 商品映射 + BOM 拆分（禮盒 → 組件）
2. 批次需求彙總與確認（含急單標記）
3. 需求扣庫存與生產建議（倍數制，急單優先）
4. 工單建立（承接現有流程）
...後續不變...
```

## Phase 1 主要風險

| # | 風險 | 嚴重性 | 緩解措施 |
|---|------|--------|----------|
| R1 | Phase 1 必須完全取代 picking-order-analyzer，完成度不足時無法上線 | 高 | 先驗證四個平台的 parser 是否可正確重構；上線前做 A/B 比對測試 |
| R2 | 庫存精度受限於每週盤點週期，累計誤差可能影響生產建議 | 中 | Phase 1 先接受此限制，但系統設計應支援更高頻盤點 |
| R3 | 主資料字典尚未完全擴充（原料細節、配方 BOM 明細、門市、倉庫） | 高 | 在 Phase 1 實作前必須先完成擴充 |
| R4 | 映射規則從既有工具遷移到正式系統時可能遺漏 | 中 | 盤點 picking-order-analyzer 現有映射規則，建立遷移清單 |
| R5 | 需求扣庫存的計算邏輯須與現行 Excel 做法對齊 | 中（已部分緩解） | Excel 統計表（`20260325-生產統計表.xlsm`）已取得並分析。三大計算邏輯已確認（BOM 拆分、安全庫存扣除、倍數制建議量）。需在 implementation spec 中建立精確公式對照 |
| R6 | 現場購買的雙模式（直售 / 預訂）作業流程尚未詳細定義 | 中 | 需補正式流程 spec |
| R7 | 安全庫存目前只在 Excel，遷移到系統需逐品確認 | 低 | 可從 Excel 批次匯入 |
| R8 | BOM 拆分邏輯複雜度 — 禮盒組合可能變動，拆分規則必須與銷售商品組成對照表對齊 | 中 | 組成對照表已有 CSV，需確認是否涵蓋所有禮盒 SKU |
| R9 | 旺季試算版本快照儲存量累積 — 每次異動保存快照，旺季期間異動頻繁 | 低 | 快照以差異壓縮為主；封存後可冷儲存 |
| R10 | 試算資料被誤用為正式依據 — 試算結果非正式帳，但操作者可能混淆 | 中 | UI 明確標示「試算 — 非正式」；系統強制試算不能觸發正式扣帳 |
