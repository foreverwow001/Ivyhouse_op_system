# Portal UI/UX Baseline

更新日期：2026-04-04

Authoritative source：是

## 目的

本文件定義 Ivyhouse OP System 在 `Idx-023` 前端 Portal 起步階段的 UI/UX 參考基線，作為後續 login page、landing page、模組入口頁、流程圖首頁與相關前端元件的共同依據。

本文件的目標不是把官網複製成後台，而是把品牌氣質、視覺語言與入口體驗轉譯成適合營運系統的 Portal 基線。

## 範圍

- login page
- login 後 landing page
- 流程圖型首頁 / 入口頁
- 模組入口卡片
- 2.5D icon 視覺語言
- 品牌色、字體、文字顏色、卡片與按鈕 style baseline

本文件不直接定義：

- 後端 auth contract
- RBAC matrix
- 正式 business state machine
- 各業務模組細部表單欄位

### 品牌視覺來源

- 官網：`https://www.ivyhousetw.com`
  - 主要特徵：奶油白底、暖金與焦糖棕點綴、深綠標題、溫暖烘焙感、柔和高級感、偏禮盒與甜點品牌氣質。

### 入口頁形式來源

- 使用者提供之流程圖參考截圖
  - 主要特徵：以節點與箭頭呈現業務主線，節點可分群，視覺中心在流程而非 KPI 儀表板。

## 核心設計原則

### 1. 品牌相似，但不做行銷站複製

- Portal 必須看起來屬於艾薇手工坊體系。
- 但 Portal 是營運系統，不應複製官網的大量 banner、商品促購語與行銷導向版型。
- 規則：沿用品牌色彩、字體氣質、材質感與圓角 / 留白語言；捨棄電商導購結構。

### 2. 入口頁優先於儀表板

- 第一版 landing page 以「流程導向入口頁」優先，不以 KPI dashboard 為主。
- 目的：讓不同角色在登入後能快速理解系統主線、找到自己的入口、理解上下游關係。

### 3. 品牌感不能破壞營運效率

- 後續模組頁可逐步提高資訊密度，但登入與 landing page 必須先建立清楚、安定、可信賴的入口感。
- 文字可有品牌氣質，但表單、清單、狀態與錯誤提示必須高度可讀。

### 4. Icon 必須成為一套語言

- icon 不應混用 flat emoji、隨機 outline icon、照片式插圖。
- 必須建立「可愛且典雅的 2.5D」統一規範，至少覆蓋：登入頁輔助插圖、landing page 模組入口、流程圖節點、空狀態。

## 視覺基線

## 色彩方向

### 主色群

- `Brand Cream`：奶油白 / 柔白，作為主要背景底色。
- `Warm Beige`：暖米杏，作為次背景、卡片層次或區塊分隔。
- `Caramel Gold`：焦糖金 / 蜂蜜金，作為主按鈕、重點線、分隔 accent、CTA。
- `Cocoa Brown`：可可棕，作為次標題、輔助 icon 深色面、強調資訊。
- `Deep Ivy Green`：深墨綠 / 森林綠，作為 H1、品牌標題、重要 heading、部份 icon 主陰影色。

### 功能色原則

- 成功、警示、錯誤、資訊色必須存在，但飽和度應低於一般科技產品預設值，避免破壞品牌溫度。
- 功能色要服從可讀性，不可因為追求品牌一致而犧牲辨識。

### 禁止事項

- 不可採用偏紫色科技漸層作為主視覺。
- 不可用純黑 / 純白高對比作為整體基調。
- 不可把整站做成冷藍 ERP 風格。

## 字體策略

### 標題

- 中文主標題應帶高級感與品牌感，適合使用襯線或具優雅筆畫對比的顯示字。
- 英文品牌輔標可採高級感 serif 或帶間距的 refined sans。

### 內文與資料區

- 表單、表格、狀態標籤、說明文與操作按鈕應使用高可讀 sans-serif。
- 規則：品牌標題字不可延伸到大量資料表與高頻表單區。

### 文字層級

- H1：品牌感強，允許較大尺寸與較高留白。
- H2 / Section title：在品牌感與可讀性之間取平衡。
- Body：以清楚、易掃描為主。
- Label / Helper / State：字級不可過小，不得用低對比灰字製造「高級感」。

## 圓角、陰影與材質感

- 卡片與大型入口塊使用中大圓角，建立溫和與親近感。
- 陰影採柔和多層次，不採極深黑影。
- 允許少量霧面光感、柔光背景、紙盒 / 烘焙包裝的材質聯想，但避免過度擬物。

## Iconography：可愛且典雅的 2.5D

## 目標語氣

- 可愛：有親和力、圓潤、易懂，不生硬。
- 典雅：不幼稚、不廉價、不過度糖果色。
- 2.5D：具備明確體積感、輕微透視與光影，而非純扁平 icon。

## 規範

- 幾何：圓角矩形、柔和轉角、適度厚度。
- 透視：保持輕微 isometric / 斜視角，不要過強透視變成 3D 遊戲物件。
- 光影：一主光源、柔和陰影、局部高光，不做金屬鏡面。
- 配色：以品牌奶油白、暖金、焦糖棕、深綠與少量功能色為主。
- 描邊：可有細描邊或邊緣陰影，但不可回到 flat outline icon。
- 細節：允許緞帶、紙盒、文件、表單、包裝、原料、報表等柔和擬物元素。

## 禁止事項

- 不可混用 emoji 當正式功能 icon。
- 不可混用扁平線框 icon 與 2.5D icon。
- 不可做成過度卡通、表情貼圖或兒童向視覺。
- 不可用過度鮮豔螢光色。

## Login page 要求

### 目的

- 建立「艾薇手工坊內部營運系統」的第一印象。
- 在品牌感、可信賴感與簡潔登入流程之間取得平衡。

### 必要元素

- 品牌標題與英文副標
- 簡潔登入表單
- 柔和品牌背景或局部插圖
- 清楚的錯誤訊息、提交中狀態、登入失敗回饋

### 建議結構

- 左右分欄或上下分區皆可，但需保有大量留白。
- 一側放品牌氣質區：標題、簡短說明、柔和 2.5D 插圖或 icon 群。
- 一側放登入表單：帳號、密碼、登入 CTA、必要輔助說明。

### UX 原則

- 不得讓登入頁看起來像一般套版 admin login。
- 不得為了品牌效果犧牲表單清晰度。
- 支援 clear focus state、鍵盤操作與錯誤提示。

## Landing page 要求

### 定位

- login 後的第一頁應為「流程導向入口頁」，不是先做傳統 KPI dashboard。

### 目的

- 讓使用者快速理解系統主線。
- 提供常用模組與流程節點的入口。
- 降低新使用者進入系統時的迷失感。

### 頁面結構

- 頂部：品牌識別、使用者資訊、必要通知與全域導航。
- 主視區：流程圖或流程卡片群，對應訂單、採購、進貨、庫存、排程、生產、出貨、報表等主線節點。
- 補充區：常用捷徑、最近作業、待處理事項，可在第二階段補入，不作為第一版核心。

### 流程圖區規則

- 以節點與關聯線呈現，不必 100% 仿照參考截圖，但要保留「一眼看懂主線」的導覽感。
- 每個節點都要有：
  - 2.5D icon
  - 明確中文名稱
  - 可選的簡短說明
  - 點擊後的入口動作
- 關聯線應輔助理解，不可比節點本身更搶眼。

### 模組入口卡片規則

- 參考簡貨單分析系統的入口卡片形式：
  - 大卡片
  - icon 置中或上方居中
  - 清楚標題
  - 一段短說明
  - 單一主要 CTA
- 卡片應保有明確 hover、focus、disabled 狀態。

## 元件基線

### 按鈕

- Primary button 使用暖金 / 焦糖金基調。
- Secondary button 可用奶油底加深綠 / 可可棕描邊。
- 所有按鈕都需有 hover、focus、disabled、loading 狀態。

### 卡片

- 卡片需有清楚層次、圓角與柔和陰影。
- 入口卡片優先使用 icon + title + description 結構。
- 不可把卡片內容塞成密集表格。

### 表單

- 表單以高可讀性為優先。
- label 必須清楚，不可只靠 placeholder。
- helper text、error text、success state 需明確。
- 登入與高頻操作頁不得使用低對比、低字級、過度細體字。

### 狀態與回饋

- loading 優先用 skeleton 或明確狀態，而非只顯示轉圈。
- error 不得 silent fail。
- empty state 需提供下一步指引。

## 系統必要需求整合

品牌與 style 要求必須與以下營運系統需求並存：

- 權限入口清楚，不可把高風險操作藏在風格化互動下。
- 角色常用工作需能快速到達。
- 後續資料密集頁面需可演進，不可被首頁 style 綁死。
- 稽核、錯誤狀態、表單驗證與流程狀態必須優先保持清楚。

## Do / Don't

### Do

- 做出帶烘焙品牌溫度的營運 Portal
- 使用奶油白、暖金、深綠、焦糖棕作為主語言
- 保留大量留白與清楚入口
- 讓流程圖首頁成為導覽，不只是裝飾
- 建立一套可延伸的 2.5D icon 規格

### Don't

- 不要做成 generic SaaS dashboard
- 不要做成官網電商頁直接移植
- 不要使用冷藍科技後台配色
- 不要混用多套 icon 風格
- 不要為了品牌感犧牲表單與狀態可讀性

## 實作前仍需補的最小前置確認

以下項目在開始前端實作前仍應先補齊，但不影響本文件作為基線：

1. 官網正式色票與字體的實際 token 抽取
2. landing page 第一版節點清單與模組映射
3. login page 所需品牌圖像資產是否已有正式版本
4. 2.5D icon 是否採自繪、生成後人工修正，或外包 / 插畫素材庫策略
