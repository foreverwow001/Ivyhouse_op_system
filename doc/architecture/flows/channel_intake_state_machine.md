# Channel Intake State Machine

更新日期：2026-03-29

Authoritative source：否（draft）

## 目的

本文件把 Intake / Mapping / Exception 三份 spec 往下落成正式狀態機草案，定義各交易 / 作業實體的：

- 狀態名稱
- 狀態語意
- 允許轉移
- 不可逆節點
- 人工介入邊界

本文件是 `unified_status_semantics.md` 在渠道 intake 領域的下層展開。

## 範圍

本版涵蓋以下 6 類狀態機：

1. 來源檔案
2. 匯入批次
3. 解析結果行
4. 映射結果行
5. Intake 例外事件
6. 映射規則提案 / 新商品提案

## 共通原則

### 1. 不可靜默跳狀態

- 任何狀態轉移都必須有事件或人工決策依據
- 不得把 `解析失敗` 直接跳成 `已忽略` 而不留痕

### 2. 正式確認後不可整批覆寫

- `匯入批次` 一旦進入 `已確認`，不得回退成 `草稿`
- 後續修正需用新事件或修正批次承接

### 3. 單批決策與永久治理分離

- `例外事件` 可被單批接受
- 但這不等於 `映射規則提案` 或 `新商品提案` 已核准發布

## 1. 來源檔案狀態機

### 狀態定義

| 狀態 | 語意 |
|------|------|
| `已上傳` | 檔案已保存，但尚未送入 parser |
| `解析中` | 正在讀檔與抽取 |
| `已解析` | 已完成結構化輸出 |
| `解析失敗` | 無法完成讀檔或抽取 |
| `已封存` | 已完成追溯保存，不再變動 |

### 允許轉移

| 來源狀態 | 目標狀態 | 條件 |
|----------|----------|------|
| `已上傳` | `解析中` | 系統開始處理 |
| `解析中` | `已解析` | parser 成功 |
| `解析中` | `解析失敗` | parser 失敗 |
| `已解析` | `已封存` | 批次關閉或封存 |
| `解析失敗` | `解析中` | 允許重新觸發解析 |

### 不允許轉移

- `已封存 -> 解析中`
- `已解析 -> 已上傳`

## 2. 匯入批次狀態機

### 狀態定義

| 狀態 | 語意 |
|------|------|
| `草稿` | 批次已建立，尚未開始完整處理 |
| `解析中` | 批次正在進行檔案解析 |
| `待映射` | 解析完成，待進行映射 |
| `待人工確認` | 映射與例外已收斂，待操作者確認 |
| `例外處理中` | 仍有未解例外，不可正式確認 |
| `已確認` | 正式需求或試算批次已成立 |
| `已作廢` | 僅限未確認前的作廢 |

### 允許轉移

| 來源狀態 | 目標狀態 | 條件 |
|----------|----------|------|
| `草稿` | `解析中` | 開始 intake |
| `解析中` | `待映射` | 解析完成 |
| `待映射` | `待人工確認` | 所有必要映射完成 |
| `待映射` | `例外處理中` | 出現未解例外 |
| `例外處理中` | `待人工確認` | 例外已完成單批決議 |
| `待人工確認` | `已確認` | 人工確認完成 |
| `草稿` | `已作廢` | 尚未啟動處理 |
| `解析中` | `已作廢` | 允許中止且未產生正式結果 |

### 不可逆節點

- `已確認` 為不可逆節點
- `已確認` 後如需修正，應建立修正批次或差異事件，不得回退本批次狀態

## 3. 解析結果行狀態機

### 狀態定義

| 狀態 | 語意 |
|------|------|
| `已抽取` | parser 已抽出原始商品與數量 |
| `待人工確認` | 解析信心不足，需要人工確認 |
| `待映射` | 可交由映射層處理 |
| `已拒絕` | 確認為無效解析結果 |

### 允許轉移

| 來源狀態 | 目標狀態 | 條件 |
|----------|----------|------|
| `已抽取` | `待映射` | 解析信心充足 |
| `已抽取` | `待人工確認` | 低信心或多義 |
| `待人工確認` | `待映射` | 人工確認通過 |
| `待人工確認` | `已拒絕` | 確認該列無效 |

## 4. 映射結果行狀態機

### 狀態定義

| 狀態 | 語意 |
|------|------|
| `待映射` | 尚未套用規則 |
| `已映射` | 已成功命中規則或人工指定 |
| `待人工覆核` | 有結果但信心不足 |
| `未映射` | 無法對應正式 SKU |
| `已接受` | 已完成人工接受，可進後續流程 |
| `已轉例外` | 已建立例外事件處理 |

### 允許轉移

| 來源狀態 | 目標狀態 | 條件 |
|----------|----------|------|
| `待映射` | `已映射` | 高信心規則命中 |
| `待映射` | `待人工覆核` | 中低信心命中 |
| `待映射` | `未映射` | 無命中 |
| `待人工覆核` | `已接受` | 人工覆核接受 |
| `待人工覆核` | `已轉例外` | 需例外處理 |
| `未映射` | `已轉例外` | 建立例外事件 |
| `已映射` | `已接受` | 可直接進正式流程 |

## 5. Intake 例外事件狀態機

### 狀態定義

| 狀態 | 語意 |
|------|------|
| `待處理` | 例外已建立，尚未有人接手 |
| `人工處理中` | 正在做單批決策或提案 |
| `單批已接受` | 只對本批次接受 |
| `已轉規則提案` | 已升級成映射規則提案 |
| `已轉商品提案` | 已升級成新商品提案 |
| `已轉工程待辦` | 屬新版型 / 新 grammar |
| `已關閉` | 本次例外流程結束 |

### 允許轉移

| 來源狀態 | 目標狀態 | 條件 |
|----------|----------|------|
| `待處理` | `人工處理中` | 有人接手 |
| `人工處理中` | `單批已接受` | 僅本批次接受 |
| `人工處理中` | `已轉規則提案` | 需補規則 |
| `人工處理中` | `已轉商品提案` | 需補商品 |
| `人工處理中` | `已轉工程待辦` | 需改 parser |
| `單批已接受` | `已關閉` | 決議已完成 |
| `已轉規則提案` | `已關閉` | 提案已建立 |
| `已轉商品提案` | `已關閉` | 提案已建立 |
| `已轉工程待辦` | `已關閉` | 工程待辦已建立 |

## 6. 提案狀態機

### 映射規則提案 / 新商品提案共用狀態

| 狀態 | 語意 |
|------|------|
| `草稿` | 已建立但尚未送審 |
| `待審核` | 等待正式治理角色判斷 |
| `已核准` | 可進入正式建檔 / 發布流程 |
| `已駁回` | 本輪不採納 |
| `已發布` | 已轉成正式規則或正式主資料 |

### 允許轉移

| 來源狀態 | 目標狀態 | 條件 |
|----------|----------|------|
| `草稿` | `待審核` | 提案提交 |
| `待審核` | `已核准` | 治理角色核定 |
| `待審核` | `已駁回` | 明確不採納 |
| `已核准` | `已發布` | 正式資料建立完成 |

## 與共通生命週期狀態的對齊

- 本文件中的交易狀態不等於主資料 `草稿 / 啟用 / 停用`
- 但 `映射規則提案`、`新商品提案` 在正式發布成主資料後，仍必須回到 `unified_status_semantics.md` 的主資料生命週期治理
