# Channel Intake Exception Resolution Spec

更新日期：2026-03-29

Authoritative source：否（draft）

## 目的

本文件定義 Ivyhouse OP System Phase 1 在渠道 intake 中對「解析失敗、低信心、未映射與版型異常」的正式例外處理流程，避免系統沿用 legacy analyzer 常見的隱性過濾或人工口頭修正模式。

本文件的核心要求是：

- 例外必須顯式留痕
- 單批修正與永久規則變更必須分離
- 人工確認結果應可回流到後續治理，而不是留在聊天或 Excel

## 適用範圍

適用於下列來源：

- MOMO / MO店+
- 蝦皮
- 官網
- 橘點子
- 後續所有接入同一 intake pipeline 的渠道

## 例外類型

### A. 平台辨識例外

適用情境：

- 檔案格式可讀，但無法判定來源平台
- 平台標記與既有 contract 不符

系統動作：

1. 建立例外事件
2. 不自動分派錯誤 parser
3. 要求人工選擇渠道或退回

### B. 解析例外

適用情境：

- 檔案無法讀取
- 表頭缺失
- 主要欄位抽取失敗
- 新 grammar 無法解析

系統動作：

1. 建立 `解析失敗` 例外
2. 保留來源檔案與錯誤訊息
3. 不得靜默丟棄

### C. 低信心解析

適用情境：

- parser 抽得到疑似商品與數量，但信心不足
- 同一列可能存在多種解讀

系統動作：

1. 產生結構化結果，但標記 `待人工確認`
2. 不得直接進正式需求

### D. 未映射例外

適用情境：

- 原始商品文字已被 parser 抽出，但無法對應正式 SKU
- 平台出現新商品、新口味、新名稱別名

系統動作：

1. 進入 mapping exception queue
2. 允許單批人工指定 SKU
3. 可提報為新商品 / 新映射規則治理需求

### E. 規則衝突例外

適用情境：

- 多條規則同時命中
- 商品與規格對到不同 SKU

系統動作：

1. 不自動挑一條默默套用
2. 進人工決策
3. 保留衝突證據

## 正式流程

```text
來源檔案 / 手動輸入
  -> parser
  -> 結構化解析結果
  -> mapping
  -> 若成功：進批次需求草稿
  -> 若失敗或低信心：進例外佇列
  -> 人工確認
  -> 單批接受 or 提報永久治理
```

## 人工處理動作

### 1. 單批接受

適用情境：

- 本次批次可由操作者明確判斷
- 但尚不足以直接發布永久規則

效果：

- 只影響本批次
- 不自動新增正式映射規則

### 2. 提報新映射規則

適用情境：

- 已確認是穩定重複出現的平台名稱別名
- 不只是一次性手修

效果：

- 建立規則提案
- 交由正式治理角色核定後生效

### 3. 提報新商品

適用情境：

- 平台上出現系統內尚不存在的正式可銷售項

效果：

- 建立新商品提案
- 待主資料治理完成後，才能成為正式可映射對象

### 4. 提報新 grammar / 新版型

適用情境：

- 問題不在商品，而在檔案結構或語法改變

效果：

- 進工程待辦
- 不把問題誤歸因為主資料不足

## 單批修正與永久治理分離原則

### 單批修正

- 作用範圍：僅本批次
- 可做內容：指定 SKU、調整數量、標示例外原因
- 不可做內容：直接覆寫正式主資料或正式規則

### 永久治理

- 作用範圍：後續批次
- 可做內容：新增商品、發布映射規則、停用舊規則、調整 BOM / 組成
- 必須留版本與核定紀錄

## 特殊案例要求

### `提袋加購`、咖啡品項

- 在目前業務判斷下，不應再被 parser 或 mapping 層直接靜默忽略
- 若尚未建檔，應進未映射例外
- 若已建檔，應允許正式映射

### `試吃:`

- 若句型符合既有 grammar，但內含新商品，屬未映射例外，不是 parser 例外
- 若句型本身不符合既有 grammar，才屬 parser 例外

### 橘點子 `.xls`

- 若檔案可被正式 intake 技術正常讀取，即視為可解析來源
- 不得僅因某個開發用函式庫嚴格報錯就直接判定來源檔異常

## 最低必要欄位

| 欄位 | 說明 |
|------|------|
| `例外事件ID` | 正式識別 |
| `匯入批次ID` | 關聯 intake |
| `來源檔案ID` | 關聯原檔 |
| `來源列識別` | 關聯原始位置 |
| `例外類型` | 平台辨識 / 解析 / 未映射 / 衝突 / 低信心 |
| `原始商品文字` | 原始輸入 |
| `原始規格文字` | 若有規格則保留 |
| `原始數量` | 原始數量 |
| `建議處理動作` | 單批接受 / 提報規則 / 提報新商品 / 工程處理 |
| `人工處理結果` | 最終處理方式 |
| `處理者` | 誰做決策 |
| `處理時間` | 何時處理 |
| `狀態` | 待處理 / 已接受 / 已轉提案 / 已關閉 |

## 驗收基準

### 驗收 1：所有失敗皆可追溯

- 任何例外都可回溯到檔案、列位與原始文字

### 驗收 2：單批修正不等於永久規則

- 人工在本批次指定 SKU，不會直接改掉正式映射規則

### 驗收 3：可形成後續治理輸入

- 例外處理完後，能轉成規則提案、新商品提案或工程待辦
