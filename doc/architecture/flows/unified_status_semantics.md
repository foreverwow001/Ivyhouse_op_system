# 統一狀態語意基線

更新日期：2026-04-03

Authoritative source：是

## 目的

本文件定義 Ivyhouse OP System 在 Phase 0 的統一狀態語意，先處理目前已治理的主資料與高風險參照實體，避免不同 CSV、文件或後續模組各自發明平行狀態。

## 範圍說明

- 本版先規範主資料 / 參照的生命週期狀態，與高風險規則表的核定狀態。
- 訂單、工單、採購、庫存、財務等完整 workflow state machine 仍待後續獨立文件補齊。
- 本文件是 `doc/architecture/flows/README.md` 與 `doc/architecture/data/master_data_dictionary.md` 的共同狀態底座。

## 狀態分類

### 1. 生命週期狀態

生命週期狀態回答的是：這筆主資料或規則資料，目前能不能被 consumer 當成正式基準引用。

統一欄位：`狀態`

允許值：`草稿`、`啟用`、`停用`

### 2. 核定狀態

核定狀態回答的是：高風險規則是否已完成可追溯的核定責任。

統一欄位：`核定狀態`

適用範圍：`單位換算規則表`、`生產_分裝_轉換扣帳規則表`

## 生命週期狀態定義

| 狀態 | 語意 | 可否供新 consumer 引用 | 最低欄位要求 |
|------|------|------------------------|--------------|
| `草稿` | 已建檔，但欄位品質、依據或下游契約尚未完整 | 否 | 至少要有正式 key、名稱與基本分類 |
| `啟用` | 已可作為正式 owner / consumer 引用基準 | 是 | `生效日` 不可空白，必要依據已齊備 |
| `停用` | 保留歷史追溯，但不再接受新引用 | 否 | `失效日`、`停用原因` 不可空白 |

## 生命週期轉移規則

| 轉移 | 是否允許 | 規則 |
|------|----------|------|
| `草稿 -> 啟用` | 允許 | 必須完成最低欄位、owner 確認與必要下游契約 |
| `草稿 -> 停用` | 允許 | 用於提案撤銷或確認不採納，仍需保留停用原因 |
| `啟用 -> 停用` | 允許 | 必須先處理 consumer 切換或明確接受既有歷史引用 |
| `停用 -> 啟用` | 非預設 | 需有正式復用理由、重寫 `生效日`、清楚處理既有 `失效日` 與審計備註 |

## 核定狀態定義

| 核定狀態 | 語意 | Phase 0 使用時機 | 必要欄位 |
|----------|------|------------------|----------|
| `暫行核定` | 已完成當前治理收斂，可作為正式基準使用，但 RBAC / approval matrix 尚未完全定稿 | 本輪四表回填預設值 | `核定日期`、`核定人` |
| `正式核定` | 已由明確角色依正式流程核定，必要時再向下追到具體人員 | 目前依 `doc/architecture/roles/README.md` 定義的第一版 RBAC / approval matrix 使用 | `核定日期`、`核定人` |
| `撤銷核定` | 原核定已被撤回，資料不得再作為正式基準 | 例外處理 | `核定日期`、備註或相對應證據 |

## 欄位語意

| 欄位 | 語意 | 規則 |
|------|------|------|
| `生效日` | 正式引用開始日 | `啟用` 不可空白 |
| `失效日` | 正式引用停止日 | `停用` 不可空白；`啟用` 應為空白 |
| `停用原因` | 停用決策與替代關係摘要 | `停用` 不可空白 |
| `來源建立方式` | 人工建立、提案落地、資料整併、治理回填等來源說明 | 全列不可空白 |
| `核定人` | 正式負責核定的角色名稱；Phase 0 第一版先不落個人姓名 | 高風險規則表不可空白，且必須對得上 `doc/architecture/roles/README.md` |
| `核定日期` | 最近一次可追溯核定日期 | 高風險規則表不可空白 |

## 2026-03-26 過渡治理規則

- 既有且已被確認可正式引用的列，若缺歷史有效期間，先以 `生效日 = 2026/3/26` 回填。
- 此處 `2026/3/26` 表示 Phase 0 治理初始化日，不代表業務上第一次生效日；若後續需要歷史追溯或版本化查詢，必須再補正式有效期間治理與版本機制。
- 在 RBAC / approval matrix 尚未建立前，曾允許以 `暫行核定` 與待補角色值作過渡回填。
- 自 `doc/architecture/roles/README.md` 已定義正式核定角色且規則表資料值完成回收後，Phase 1 正式基線應一律使用 `正式核定`，不得再保留待補角色字串。

## 過渡治理結束條件

- 自 Idx-005 第一版 RBAC / approval matrix 建立後，新增或修改的高風險規則不得再使用 `暫行核定` 與待補角色字串。
- 目前正式治理基線已假設既有可用規則都已回收為 `正式核定`；若仍有歷史列保留過渡值，僅可作為歷史痕跡，不得作為新流程的正式基準。

## Phase 1 最小狀態銜接

- `daily ops` 主線目前只正式收斂到：主資料 / 規則使用本文件的生命週期與核定欄位語意，daily inventory / production planning / replenishment 再引用各自流程文件定義事件與提醒。
- `inventory-count` 對人工盤點型出貨用品採 `DRAFT -> IN_PROGRESS -> COMPLETED / CANCELLED` session 模型；其中 opening balance 允許 `CANCELLED` 作為中斷補救，但 cancelled session 不得寫入正式差異基線。
- `production-planning`、`end-of-day replenishment` 與一般日常扣帳流程，現階段仍以各自 flows 規格的事件語意為準，不在本文件假裝已有完整全域 state machine。
- 後續若要支援個人層級核定、歷史回溯與版本查詢，應在 RBAC matrix 與流程 state machine 擴版時一併定義。

## 與後續 workflow state machine 的關係

- 本文件處理的是共通語意，不等於訂單、工單、採購、庫存、財務流程的完整狀態機。
- 後續各流程 state machine 必須引用本文件的欄位語意與不可逆原則，但可以在各自領域定義更細的業務狀態。
