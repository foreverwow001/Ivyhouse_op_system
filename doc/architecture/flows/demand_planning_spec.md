# 旺季試算與需求規劃規格草案

更新日期：2026-03-29

Authoritative source：否（draft）

## 目的

本文件定義 Ivyhouse OP System Phase 1 的「旺季試算與需求規劃」支援性能力的業務規格。此能力與正式營運主線並存但邏輯分離：試算不直接觸發正式扣庫存、不自動產生正式工單，但與正式主線共用同一套主資料、BOM 拆分與單位換算。

## 業務背景

### 為什麼需要試算

- 糕點行業有明確的旺季（中秋節、過年），旺季需求可達淡季的數倍
- 旺季客人會指定出貨日期，指定日密集時可能單週需求暴增
- 目前產能無法在單日內滿足旺季每日需求
- 需要提前 1 到 4 週規劃，把未來的生產工作量提前分配到未來數日
- 目前以 Excel 手工做此規劃，旺季依賴度極高

### 試算與正式的根本差異

| 面向 | 試算規劃 | 正式營運 |
|------|----------|----------|
| 目的 | 預看未來壓力、提前排產能 | 處理已確認要做、要扣帳、要出貨的需求 |
| 時間範圍 | 未來 1 到 4 週（以日為最小單位） | 當日或已確認的近日需求 |
| 資料來源 | 所有平台的需求量皆可納入；已有指定出貨日者沿用，無指定出貨日者由操作者分配到規劃日期 | 當日各平台撿貨單、手動補單、現場需求 |
| 資料變動 | 每天持續追加、修正、取消 | 人工確認後才成立，成立後走修正批次 |
| 對正式帳的影響 | 無。不改正式庫存、不產生正式工單 | 會。扣庫存、生產建議、工單、出貨追溯 |
| 歷史版本 | 保留。需回溯某次規劃的原始估量與後續異動 | 保留。主線追溯鏈 |
| 使用者 | 操作者規劃 + 給主管看；不需正式簽核 | 操作者確認 + 系統正式落帳 |

### 試算不串進正式主線的原因

實際每日生產量的決定，除了預估的指定日期需求外，還必須加上：

- 當天各平台進來的日常訂單
- 不同平台各自的出貨時效要求（例如蝦皮隔日到貨）
- 現貨庫存狀況
- 人工判斷（經驗、彈性、優先順序）

目前因為沒有平台 API key，這些因素無法由系統自動收斂。所以試算只能作為人工排產決策的參考輸入之一，不能被當成正式流程的前置狀態。

## 核心概念

### 規劃期間（Planning Window）

| 項目 | 定義 |
|------|------|
| 最小單位 | 日 |
| 典型範圍 | 未來 1 到 4 週 |
| 使用方式 | 操作者可建立一個規劃期間，指定起訖日。可同時存在多個規劃期間 |
| 視圖 | 可看指定期間內的每日需求分布、單週小計、跨週總量 |

### 試算版本（Planning Version）

| 項目 | 定義 |
|------|------|
| 定義 | 在同一個規劃期間內，每次實質修改都產生一個新版本快照 |
| 版本觸發 | 追加需求、扣減需求、修改商品或數量時，系統自動保存版本 |
| 版本內容 | 該規劃期間在本版本下，所有商品 × 需求日期 × 數量的完整快照 |
| 歷史保留 | 所有版本永久保留，可回查任何歷史版本 |
| 最新版本 | 系統預設顯示最新版本，可手動切換到歷史版本查看 |

### 試算異動（Planning Adjustment）

| 異動類型 | 說明 |
|----------|------|
| 新增需求 | 匯入或手動輸入新的需求。若來源沒有指定出貨日，則由操作者指定需求日期 |
| 追加需求 | 對已存在的規劃期間追加新訂單 |
| 扣減需求 | 指定日期訂單取消或減量 |
| 修改商品或數量 | 更正匯入錯誤或客戶修改 |

每筆異動記錄：操作者、時間、異動類型、異動前後差異。

## 功能規格

### F1. 規劃期間管理

| 項目 | 說明 |
|------|------|
| 建立 | 操作者可建立規劃期間，指定起訖日（以日計） |
| 多期間並存 | 可同時存在多個規劃期間（例如第 1 週、第 2 週、第 1-4 週全覽） |
| 狀態 | 進行中 / 已封存。封存表示該期間已結束，不再接受異動 |

### F2. 需求匯入（試算專用）

| 項目 | 說明 |
|------|------|
| 來源 | 所有平台的需求量皆可納入。已有指定出貨日者直接使用；沒有指定出貨日的平台需求，可由操作者分配到規劃期間中的某一日 |
| 匯入方式 | 與正式 Intake 共用同一套匯入能力（檔案上傳、手動輸入），但匯入目標明確選擇「試算規劃」而非「正式需求」 |
| 日期基準 | 建立規劃期間時，操作者必須指定一個日期區間。明細若已有指定出貨日則沿用；若來源沒有指定出貨日，則由操作者指定該需求落在哪一日 |
| 商品映射 | 共用正式的商品映射規則與 SKU |
| BOM 拆分 | 自動執行。禮盒需求拆分為內包裝完成品組件，與正式流程一致 |
| 單位換算 | 共用正式的單位換算規則 |

### F3. 追加與扣減

| 項目 | 說明 |
|------|------|
| 追加 | 匯入新需求到既有規劃期間，數量累加 |
| 扣減 | 指定某商品在某出貨日的數量減少（因訂單取消或修改）。扣減不得使該商品該日數量低於 0 |
| 修改 | 更正已匯入的商品或數量 |
| 版本 | 每次追加 / 扣減 / 修改操作，自動保存新版本快照 |

### F4. 試算庫存推演

| 項目 | 說明 |
|------|------|
| 起始庫存 | 取正式庫存台帳的當前可用庫存作為起點 |
| 推演邏輯 | 預設以試算版本自身需求做逐日推演。正式已確認需求僅作唯讀疊加參考，不與試算需求自動扣抵或合併 |
| 安全庫存警示 | 若某日預估庫存餘量低於安全庫存，標示警告 |
| 缺口提示 | 若某日預估庫存餘量為負，標示缺口數量 |
| 限制 | 推演不考慮未來的生產入庫（因為還沒排工單），所以預估值偏保守。操作者可自行判斷是否提前排產 |
| 庫存粒度 | 庫存推演以內包裝完成品層級為主；對直接以銷售商品層級管理的品項，可保留銷售商品層級 |

### F5. 視圖與呈現

| 視圖 | 說明 |
|------|------|
| 每日需求分布 | 在規劃期間內，每日的商品 × 數量矩陣 |
| 單週小計 | 以週為單位彙總需求量 |
| 跨週總量 | 整個規劃期間（最長 4 週）的總需求量 |
| 每日預估庫存 | 逐日庫存推演結果 |
| 缺口與警告列表 | 列出所有預估低於安全庫存或為負的品項 × 日期 |
| 版本比對 | 可選擇兩個歷史版本並排比對差異 |

### F6. 版本歷史

| 項目 | 說明 |
|------|------|
| 自動版本 | 每次異動自動保存 |
| 版本瀏覽 | 可查看任何歷史版本的完整規劃快照 |
| 版本比對 | 可比較兩個版本間的差異：新增、刪除、數量變化 |
| 不可刪除 | 歷史版本不可刪除，保留完整規劃決策軌跡 |

### F7. 分享

| 項目 | 說明 |
|------|------|
| 存取層級 | 行政、會計、主管三個角色均可查看與操作 |
| 分享方式 | 規劃期間的當前版本可直接由主管在系統中查看，不需額外匯出 |
| 不需正式簽核 | 試算是規劃工具，不是正式審批流程 |

## 與正式營運主線的關係

### 邊界規則

| 規則 | 說明 |
|------|------|
| 不改正式庫存 | 試算的需求不會從正式庫存台帳中扣減任何數量 |
| 不產生正式工單 | 試算不能直接變成正式工單 |
| 不取代人工決策 | 每日實際生產量仍由人工決定，試算只提供參考 |
| 不自動對沖正式需求 | 同商品同日期在試算與正式可並存；系統不自動視為同一筆需求 |
| 共用主資料 | 商品、BOM、單位換算、渠道、映射規則全部共用正式主資料 |
| 共用匯入能力 | 試算和正式共用同一套檔案上傳與解析基建，分流點在匯入目標選擇 |

### 規劃期間重疊規則

| 規則 | 說明 |
|------|------|
| 可重疊存在 | 同一日期可存在多個規劃期間，用於不同視角或不同規劃版本 |
| 不做全域合計 | 系統預設只在單一規劃期間內做統計與推演，不把多個規劃期間自動合併成全域總量 |
| 重疊警示 | 若同商品同日期在多個規劃期間同時存在，系統應提示可能重複解讀 |

### 試算對人工排產決策的貢獻

操作者或主管在做每日生產決策時，可參考的輸入：

| 輸入 | 來源 |
|------|------|
| 今天正式需求 | 正式營運主線（能力 A/B） |
| 未來 1-4 週壓力 | 旺季試算工作區 |
| 當前庫存 | 正式庫存台帳 |
| 安全庫存 | 主資料設定 |
| 平台時效要求 | 人工經驗（蝦皮隔日到貨等） |
| 生產產能 | 人工經驗 |

最終生產量 = 人工綜合判斷，不是系統自動計算。

## 資料模型概覽

### 新增實體

| 實體 | Owner | 說明 |
|------|-------|------|
| `規劃期間` | Production / Planning | 一個旺季試算的時間框架。有起訖日、狀態、建立者 |
| `規劃版本` | Production / Planning | 規劃期間下的版本快照。自動遞增，結構不可變、不可刪 |
| `規劃需求行` | Production / Planning | 版本下的需求明細：商品 × 需求日期 × 數量 × BOM 拆分後組件需求 |
| `規劃異動` | Production / Planning | 每次追加 / 扣減 / 修改的操作紀錄。關聯到異動前後的版本 |
| `規劃庫存推演` | Production / Planning | 以版本為基準的逐日庫存推演結果快照。不落正式庫存台帳 |

### 共用既有實體（不新增，只讀取）

| 實體 | 使用方式 |
|------|----------|
| 銷售商品主檔 | 需求行的商品引用 |
| 內包裝完成品主檔 | BOM 拆分後的組件層級需求與庫存推演 |
| 銷售商品組成對照表 | BOM 拆分依據 |
| 單位換算規則 | 試算中的單位呈現 |
| 庫存台帳（唯讀） | 推演起始庫存 |
| 安全庫存設定 | 缺口與警告判斷 |
| 商品映射規則 | 匯入時商品映射 |

## Phase 1 驗收基線（支援性能力）

此能力列為 Phase 1 支援性能力，不納入正式主線驗收（V1-V16），但作為獨立驗收項目。

| # | 驗收項目 | 說明 |
|---|----------|------|
| S1 | 可建立規劃期間 | 指定起訖日（以日計），建立成功 |
| S2 | 可匯入需求到規劃期間 | 共用正式匯入能力，目標選擇試算 |
| S3 | 可追加 / 扣減需求 | 追加與扣減操作正確累加或累減 |
| S4 | BOM 自動拆分 | 禮盒需求自動拆分為組件需求 |
| S5 | 版本自動保存 | 每次異動產生新版本快照 |
| S6 | 可查看歷史版本 | 可切換歷史版本查看 |
| S7 | 可比對兩個版本差異 | 比對差異正確呈現 |
| S8 | 每日需求分布可查看 | 規劃期間內逐日商品需求矩陣 |
| S9 | 單週小計與跨週總量 | 按週彙總與跨週總量正確 |
| S10 | 庫存推演可查看 | 逐日預估庫存餘量正確 |
| S11 | 缺口與安全庫存警告 | 低於安全庫存或負庫存正確標示 |
| S12 | 不影響正式庫存 | 所有試算操作不改正式庫存台帳 |
| S13 | 不產生正式工單 | 試算不能直接轉為工單 |
| S14 | 主管可查看 | 三個角色均可存取 |

## 風險

| # | 風險 | 嚴重性 | 緩解措施 |
|---|------|--------|----------|
| SP1 | 版本快照儲存量大 | 低 | 快照以差異壓縮為主，非全量複製；封存後可冷儲存 |
| SP2 | 試算推演不含未來生產入庫，偏保守 | 中 | 在推演視圖明確標示「不含未排工單的預期入庫」，操作者自行判斷 |
| SP3 | 試算資料如果被誤用為正式依據 | 中 | UI 明確標示「試算 — 非正式」；系統強制試算不能觸發正式扣帳 |
| SP4 | 跨期間重疊時需求可能被重複解讀 | 中 | 系統預設只在單一規劃期間內統計；若同商品同日期跨期間重疊，顯示提醒 |
