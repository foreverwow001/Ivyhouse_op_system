# Intake / Demand Aggregation 流程規格草案

更新日期：2026-03-29

Authoritative source：否（draft）

## 目的

本文件定義 Ivyhouse OP System Phase 1 的「需求匯入與彙總」（Intake / Demand Aggregation）正式流程規格。此流程是 Phase 1 核心營運主線的第一段，負責將每日多來源的銷售需求收斂為一份可交付後續流程（需求扣庫存 → 生產建議 → 工單）的「已確認批次需求」。

本流程也是旺季試算工作區的共用匯入基建——兩者共用檔案上傳、解析、商品映射與 BOM 拆分引擎，但在「匯入目標選擇」這一步分流：正式需求進入主線、試算需求進入旺季試算工作區。

下層落地契約請同步參照：

- `doc/architecture/data/channel_intake_entity_field_contract.md`
- `doc/architecture/flows/channel_intake_state_machine.md`
- `doc/architecture/flows/channel_intake_api_contract.md`
- `doc/architecture/flows/channel_intake_sample_acceptance_matrix.md`

## 業務背景

### 現況

| 現行工具 | 角色 |
|----------|------|
| picking-order-analyzer | 前端 JS/HTML，解析蝦皮/MO店+/官網/橘點子的撿貨單，產出商品 × 數量統計 |
| Excel 生產統計表 | 手工匯入 picking-order-analyzer 需求 + 其他來源需求 → BOM 拆分 → 當日需求彙總 |
| 人工判斷 | 操作者看統計表、看庫存、手動決定今天要做什麼 |
| 鼎新 A1 | 僅用於蝦皮撿貨單下載，不是 ERP |

### Phase 1 目標

Phase 1 將 picking-order-analyzer 的解析能力與 Excel 統計表的彙總能力整合為一條完整的系統化流程，並加上持久化、追溯與審計。Phase 1 上線後 picking-order-analyzer 直接停用。

### 渠道欄位規則

| 規則 | 說明 |
|------|------|
| 欄位原則 | 除批次層的匯入目標與來源渠道外，明細欄位以選填為原則；但在確認前仍須補齊足以識別商品與數量的最小資訊 |
| 共通欄位 | 來源渠道、商品（名稱或 SKU）、數量 |
| 選填欄位 | 指定出貨日、急單標記、備註 |
| 指定出貨日 | 電話/官網/LINE/FB 較常有指定出貨日；蝦皮等電商平台通常為當日或隔日 |

### 蝦皮特殊時效規則

| 規則 | 說明 |
|------|------|
| 隔日到貨 | 蝦皮平台規定：當天下午 2 點前的訂單，必須在晚上 7 點前寄出 |
| 切批建議 | 建議 AM 批次 + 2PM 截止批次。但目前沒有固定切批規則，由操作者彈性處理 |
| 急單含義 | 若蝦皮訂單在 2PM 前匯入，該批次隱含「今日必出」的急迫性 |
| 擴充原則 | 目前只有蝦皮有明確 cutoff 規則，但系統設計應保留「渠道時效 / cutoff 規則」可配置化，避免未來其他平台跟進時要重改流程 |

## 核心流程

### 正式需求主線

```
┌────────────────────────────────┐
│ Step 0：檔案上傳 / 手動輸入     │
│ 操作者選擇來源渠道 + 匯入目標   │
│ （正式需求 or 旺季試算）        │
└───────────┬────────────────────┘
            │
            ▼ 匯入目標 = 正式需求
┌────────────────────────────────┐
│ Step 1：解析                    │
│ 自動解析：PDF/Excel → 原始行    │
│ 手動：結構化輸入 → 原始行       │
└───────────┬────────────────────┘
            │
            ▼
┌────────────────────────────────┐
│ Step 2：商品映射                │
│ 原始商品名稱 → 正式 SKU        │
│ 自動映射 / 低信心 → 人工覆核    │
└───────────┬────────────────────┘
            │
            ▼
┌────────────────────────────────┐
│ Step 3：BOM 拆分               │
│ 禮盒 SKU → 組件明細            │
│ 非禮盒直接透傳                  │
└───────────┬────────────────────┘
            │
            ▼
┌────────────────────────────────┐
│ Step 4：批次需求彙總草稿        │
│ 商品 × 數量 × 來源 × 出貨日    │
│ 含急單標記                      │
│ 呈現 BOM 拆分前後對照           │
└───────────┬────────────────────┘
            │
            ▼
┌────────────────────────────────┐
│ Step 5：人工確認                │
│ 操作者檢視彙總草稿              │
│ 可調整數量、修正商品、追加急單   │
│ 確認後 → 正式批次需求成立       │
└───────────┬────────────────────┘
            │
            ▼
┌────────────────────────────────┐
│ Step 6：交付後續流程             │
│ → 能力 B：需求扣庫存 → 生產建議 │
└────────────────────────────────┘
```

### 試算分流

若 Step 0 選擇「旺季試算」：

- 解析（Step 1）、商品映射（Step 2）、BOM 拆分（Step 3）照常執行
- 操作者必須先指定一個規劃日期區間；明細若無指定出貨日，需在匯入時指定其落在哪一日
- 彙總結果進入旺季試算工作區的規劃期間（見 `demand_planning_spec.md`）
- 不進入正式批次需求、不觸發需求扣庫存、不產生正式工單

## 詳細步驟規格

### Step 0：檔案上傳 / 手動輸入

| 項目 | 說明 |
|------|------|
| 操作者 | 行政、會計、主管（三個角色均可操作） |
| 匯入目標選擇 | 正式需求 / 旺季試算。默認為正式需求 |
| 來源渠道選擇 | 從銷售渠道主資料中選擇。必填 |
| 試算日期區間 | 若匯入目標為旺季試算，必須指定一個規劃日期區間 |
| 自動解析渠道 | 上傳檔案（PDF 或 Excel）。系統根據渠道觸發對應 parser |
| 半結構化渠道 | 上傳報表或直接在表格中輸入 |
| 純手動渠道 | 在表格中直接輸入商品、數量、出貨日、備註等 |
| 同一天多批 | 一天可執行多次匯入，每次產生獨立的匯入批次 |
| 來源檔案保存 | 上傳的檔案保存副本，不可修改、不可刪除 |

### Step 1：解析

| 項目 | 說明 |
|------|------|
| 自動解析 | 根據渠道觸發對應 parser（蝦皮 PDF parser、MO店+ Excel parser、官網 Excel parser、橘點子 Excel parser）。解析結果為原始行列表：原始商品名稱、數量、備註 |
| Parser 來源 | 重構 picking-order-analyzer 的 ParserFactory 與各平台 parser，但正式邊界以 `channel_intake_parser_contract.md` 為準 |
| 解析失敗處理 | 若檔案格式無法辨識或解析失敗，記錄錯誤並通知操作者。不自動丟棄 |
| 手動輸入 | 直接產生原始行，不經 parser |
| 產出 | `解析結果行`：每行包含原始商品名稱、原始數量、來源渠道、備註 |

### Step 2：商品映射

| 項目 | 說明 |
|------|------|
| 自動映射 | 使用商品映射規則表，將原始商品名稱/規格映射到正式銷售商品 SKU |
| 映射信心 | 每筆映射帶信心等級（完全匹配 / 模糊匹配 / 未匹配） |
| 人工覆核 | 模糊匹配及未匹配的項目，必須由操作者覆核確認。此覆核是「本批次映射結果確認」，不是修改主資料規則 |
| 新商品處理 | 若出現完全無法映射的商品，操作者可先為該批次手動指定 SKU 或暫時標記待處理；這不等於永久新增映射規則。若只是新增商品、平台名稱別名或組合內容，應優先走資料治理，不應預設改 parser code |
| 規則維護 | 映射規則屬 Master Data owner 維護範圍。Intake 操作者可提報映射建議；實際新增、修改、停用需由受權的主資料角色執行並留痕。正式治理邊界以 `channel_product_mapping_governance.md` 與 `channel_intake_exception_resolution_spec.md` 為準 |
| 產出 | `映射結果行`：原始行 + 正式 SKU + 映射方式（自動/人工）+ 覆核狀態 |

### Step 3：BOM 拆分

| 項目 | 說明 |
|------|------|
| 拆分依據 | 銷售商品組成對照表（`1-2026-03-24_銷售商品組成對照表_template.csv`） |
| 禮盒類商品 | 自動拆分為內包裝完成品層級的組件需求。例：綜花餅乾禮盒 → 各口味餅乾 × 對應片數 |
| 非禮盒商品 | 直接透傳，不拆分 |
| 原始保留 | 拆分前的禮盒需求行與拆分後的組件需求行同時保留，可雙向追溯 |
| 單位換算 | 拆分過程中自動套用單位換算規則 |
| 產出 | 拆分後的需求行：內包裝完成品 SKU × 數量（作業單位）。保留與原始禮盒需求行的關聯 |

### Step 4：批次需求彙總草稿

| 項目 | 說明 |
|------|------|
| 彙總範圍 | 同一匯入批次內所有來源的映射 + BOM 拆分結果 |
| 呈現方式 | 商品 × 數量 × 來源渠道 × 需求日期。正式需求使用指定出貨日；旺季試算則使用指定出貨日或操作者指定的落點日期。BOM 拆分前後對照 |
| 急單標記 | 操作者可在此步驟標記特定需求為急單（當天必出貨） |
| 旺季試算口徑 | 試算時可把所有平台數量納入同一規劃期間，不以是否有指定出貨日作為是否可納入的限制 |
| 庫存參考 | 彙總畫面同時顯示該商品的當前可用庫存（唯讀參考），幫助操作者判斷 |
| 庫存檢查粒度 | 混合式：部分商品以銷售商品層級檢查、部分以內包裝完成品層級檢查，取決於商品類型 |
| 草稿可編輯 | 操作者可在此步驟調整數量、修正商品、增刪行 |
| 尚未正式 | 草稿狀態的彙總不影響正式庫存、不觸發後續流程 |

### Step 5：人工確認

| 項目 | 說明 |
|------|------|
| 確認動作 | 操作者點擊「確認」後，該匯入批次的需求正式成立 |
| 確認後不可逆 | 確認後不可整批撤回。若需修改，走修正批次流程（見訂單修正） |
| 確認紀錄 | 記錄確認者、確認時間 |
| 產出 | `批次需求彙總`：正式的商品 × 數量 × 來源 × 出貨日 × 急單標記。可交付後續流程 |

### Step 6：交付後續流程

| 項目 | 說明 |
|------|------|
| 主線 | 已確認的批次需求進入能力 B（需求扣庫存 → 生產建議） |
| 現場直售 | 現場直售需求不經生產排程，直接走現貨直售流程（能力 D3）：確認 → 直接扣庫存 → 出貨 |
| 時間觸發 | 系統不自動觸發。操作者在每批確認後，手動進入後續流程。未來可加自動觸發 |

## 訂單修正流程

### 修正類型

| 類型 | 說明 |
|------|------|
| 加單 | 追加新的需求行到已確認的批次，或建立新的修正批次 |
| 修改 | 更改已確認需求的商品規格或數量 |
| 取消 | 取消特定需求行。但不減少已排定的生產工作量 |

### 修正規則

| 規則 | 說明 |
|------|------|
| 不減已排產量 | 取消訂單時，已經排入工單的生產數量不因取消而減少。理由：短保商品一旦排產，減產不如照做再另行消化 |
| 修正批次 | 每次修正產生獨立的修正批次紀錄，含操作者、時間、修正類型、修正前後差異 |
| 追溯 | 修正批次可追溯到原始批次需求 |
| 在庫存反映 | 加單：增加需求量 → 下次需求扣庫存計算時含入。取消：若尚未排產，需求量扣減；若已排產，需求標記取消但生產量不變 |

### 修正操作者

| 角色 | 權限 |
|------|------|
| 行政 | 可操作加單、修改、取消 |
| 會計 | 可操作加單、修改、取消 |
| 主管 | 可操作加單、修改、取消 |

三個角色權限相同。Phase 1 不做差異化權限。

## 批次切分

### 規則

| 項目 | 說明 |
|------|------|
| 無固定切批規則 | 操作者自行決定何時匯入新批次。沒有系統強制的切批時間點 |
| 一天多批 | 同一天可匯入任意數量的批次。每批獨立可辨識 |
| 蝦皮建議切批 | 建議在 2PM 前做一批（涵蓋必須今日出貨的訂單），2PM 後做第二批。但不強制 |
| 當日總覽 | 系統支援查看當日所有批次的累計需求 |
| 跨日彙總 | 若有指定出貨日的需求，可按指定出貨日彙總查看 |

## 狀態模型

### 匯入批次狀態

```
草稿（上傳中 / 解析中 / 映射中）
  │
  ├── 解析失敗 → 操作者介入修正 → 回到草稿
  │
  ▼
彙總草稿（已解析 + 已映射 + 已 BOM 拆分，待人工確認）
  │
  ├── 操作者調整 → 仍在彙總草稿
  │
  ▼
已確認（人工確認後，正式成立）
  │
  ├── 修正批次 → 關聯到此批次
  │
  ▼
已交付後續（已進入需求扣庫存或後續流程）
  │
  ▼
已完成（對應出貨全部完成）
```

### 批次需求行狀態

```
草稿
  │
  ▼
已確認
  │
  ├── 已排入生產建議
  │     │
  │     ├── 已排入工單
  │     │     │
  │     │     ├── 生產中
  │     │     │
  │     │     ▼
  │     │   已完工
  │     │     │
  │     │     ▼
  │     │   已包裝
  │     │     │
  │     │     ▼
  │     │   已出貨（完成）
  │     │
  │     └── 修正（加單 / 改品 / 取消）
  │
  └── 現場直售已出貨（直售渠道專用）
```

## 與其他能力的介面

### 共用主資料（唯讀引用）

| 主資料 | Intake 使用方式 |
|--------|----------------|
| 銷售渠道 | Step 0 的渠道選擇清單 |
| 銷售商品主檔 | Step 2 映射目標 |
| 商品映射規則 | Step 2 自動映射依據 |
| 銷售商品組成對照表 | Step 3 BOM 拆分依據 |
| 內包裝完成品主檔 | Step 3 拆分後的組件層級 |
| 單位換算規則 | Step 3 拆分過程中的單位換算 |
| 庫存台帳 | Step 4 庫存參考（唯讀） |

### 輸出到後續流程

| 輸出 | 接收方 | 說明 |
|------|--------|------|
| 已確認批次需求（正式） | 能力 B：需求扣庫存 | 正式需求進入扣庫存計算 |
| 已確認規劃需求（試算） | 能力 G：旺季試算工作區 | 試算需求進入規劃期間與版本快照，不形成正式批次需求 |
| 現場直售需求 | 能力 D3：現場購買直售 | 跳過生產排程，直接扣庫存出貨 |

### 修正通知

| 事件 | 影響 |
|------|------|
| 加單確認 | 下次需求扣庫存計算時含入新增需求 |
| 取消（未排產） | 批次需求行標記取消；扣庫存計算排除 |
| 取消（已排產） | 需求行標記取消；已排工單不受影響 |
| 修改商品/數量 | 產生差異紀錄；未排產部分影響下次扣庫存計算 |

| 內包裝完成品主檔 | BOM 拆分後引用組件 SKU |
| 商品映射規則 | 映射結果行引用使用的映射規則 |
| 銷售商品組成對照表 | BOM 拆分引用組成規則 |

## 異常場景

| # | 場景 | 處理 |
|---|------|------|
| EX1 | 檔案格式無法辨識 | 記錄錯誤，通知操作者。操作者可重新上傳或手動輸入 |
| EX2 | 商品映射完全未匹配 | 標記為「待處理」。操作者可手動指定 SKU、暫時跳過、或新建映射規則 |
| EX3 | BOM 組成表缺少某禮盒的拆分規則 | 標記警告。操作者可手動輸入組件明細或暫時以禮盒整品處理 |
| EX4 | 同一批次中重複商品 | 系統自動合併同商品數量，顯示合併前後對照 |
| EX5 | 匯入後發現數量錯誤 | 若尚未確認：直接在草稿中修改。若已確認：走修正批次流程 |
| EX6 | 確認後才發現商品映射錯誤 | 走修正批次流程：原需求行取消 + 新增正確的需求行 |

## 風險

| # | 風險 | 嚴重性 | 緩解措施 |
|---|------|--------|----------|
| IF1 | Parser 重構品質風險 — 從前端 JS 搬到後端 NestJS，可能引入解析差異 | 高 | 上線前 A/B 比對測試（新系統 vs picking-order-analyzer） |
| IF2 | 映射規則遷移遺漏 — picking-order-analyzer 現有映射規則可能不完整 | 中 | 盤點所有現有映射規則，建立遷移清單並逐條驗證 |
| IF3 | BOM 組成表不完整 — 部分禮盒可能缺少拆分規則 | 中 | 與營運團隊逐一確認所有禮盒 SKU 的組成 |
| IF4 | 蝦皮 2PM 切批未自動化 — 若操作者忘記切批，可能延誤急單 | 低 | Phase 1 以人工切批為主；可加提醒通知 |
| IF5 | 修正批次追溯複雜 — 多次修正後追溯鏈可能變長 | 低 | 每次修正都記錄完整的前後差異與時間戳 |
