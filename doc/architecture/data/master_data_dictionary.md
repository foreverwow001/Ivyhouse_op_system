# Master Data Dictionary

更新日期：2026-04-03

Authoritative source：是

## 目的

本文件是 Ivyhouse OP System 在 Phase 0 的第一版主資料字典。它先收斂目前已正式治理的 CSV 主資料與高風險參照實體，並補上原料、配方版本與出貨用品 / 包裝耗材的最低可執行定義，明確定義 owner、正式 key、最低必要欄位、生命週期欄位與 consumer 邊界，避免 CSV 只是資料草稿而不是可執行契約。

## 範圍說明

- 本版先涵蓋十一類資料：銷售商品、包材主檔、內包裝完成品、單位換算規則、轉換扣帳規則、原料、配方版本、出貨用品 / 包裝耗材、銷售渠道、商品映射規則、渠道時效 / cutoff 規則。
- 本版不是 Phase 0 全量主資料字典終稿；門市、倉庫、供應商、客戶、稅別與財務參照仍待後續補齊。
- `單位換算規則` 與 `轉換扣帳規則` 雖屬高風險參照，不是傳統主檔，但因其控制數量語意與跨模組扣帳一致性，因此納入本版字典。
- `配方版本` 與 `出貨用品 / 包裝耗材` 不是本輪 CSV 治理主體，但因其直接影響生產、庫存、成本與出貨作業是否能進 ERP，所以在本版先補最低必要治理結構。
- `銷售渠道`、`商品映射規則` 與 `渠道時效 / cutoff 規則` 雖不全是傳統主檔，但它們會直接控制 Batch Intake、旺季試算、急單判定與日常訂單處理，因此本版一併納入正式治理範圍。

## 共通欄位基線

### 主資料 / 參照共通生命週期欄位

| 欄位 | 用途 | 規則 |
|------|------|------|
| `狀態` | 控制該列是否可供 consumer 正式引用 | 統一採 `草稿`、`啟用`、`停用` |
| `生效日` | 定義該列開始可被正式引用的日期 | `啟用` 不可空白 |
| `失效日` | 定義該列停止作為正式引用基準的日期 | `停用` 不可空白；`啟用` 應為空白 |
| `停用原因` | 保留停用決策與替代關係的最小審計線索 | `停用` 不可空白 |
| `來源建立方式` | 說明該列是人工建立、提案落地、資料整併或治理回填 | 不可空白 |

### 高風險規則表附加核定欄位

| 欄位 | 適用表 | 用途 | 規則 |
|------|--------|------|------|
| `核定狀態` | 單位換算規則表、轉換扣帳規則表 | 區分歷史過渡核定與正式核定 | Phase 0 完成基線以 `正式核定` 為準；`暫行核定` 只保留歷史過渡語意 |
| `核定人` | 單位換算規則表、轉換扣帳規則表 | 指向負責最終核定的角色名稱；Phase 0 第一版先不追到個人姓名 | 必須對得上 RBAC / approval matrix 的正式角色值，不得保留待補字串 |
| `核定日期` | 單位換算規則表、轉換扣帳規則表 | 保留最近一次核定基準日期 | `核定狀態` 非空時不可空白 |

### 本輪過渡回填基線

- 治理生效日：`2026/3/26`
- 已在本輪被確認可正式引用的既有資料，若缺歷史有效期間，先以 `生效日 = 2026/3/26` 回填。
- `2026/3/26` 仍是治理初始化日，但自 Idx-005 第一版 RBAC / approval matrix 建立後，高風險規則的正式基線已回收為 `核定狀態 = 正式核定`、`核定日期 = 2026/3/26` 與對應正式角色值。

| 補充規格 | `doc/architecture/data/README.md` 原料治理要求 |

治理重點：

- 原料必須先成為正式主資料，才能被採購、生產、配方與庫存引用。
- 若採購、庫存、生產單位不同，必須存在正式換算規則與有效期間，不得靠自由文字或備註處理。
- 涉及批次或效期追蹤的原料，必須在主資料層明確標記，不可交由流程默認。

第一版最低可執行補強：

- 本版正式收斂原料的最低 owner contract：`原料代碼`、正式名稱、原料類型、採購 / 庫存 / 生產單位、批次 / 效期追蹤旗標與生命週期欄位。
- `銷售商品組成對照表` 若經核定使用「原料直接分裝 / no_inner_stage」例外，引用的仍必須是本主檔已存在的正式 `原料代碼`，不得由 consumer 反向創建原料 key。
- 原料採購條件、供應商映射、交期與批次 / 效期細節仍留待後續專屬規格，不屬本版完成定義。

### 7. 配方版本

| 項目 | 定義 |
|------|------|
| 實體名稱 | 配方版本 |
| Owner module | Master Data |
| Authoritative source | 本文件第一版定義；後續應拆成獨立配方 / BOM 規格 |
| 正式 key | 組合鍵：`配方代碼 + 版本號` |
| 主要 consumer | Production、Inventory、成本計算、替代料決策 |
| 最低必要欄位 | `配方代碼`、`版本號`、`對應產品或半成品`、`生效日`、`失效日`、`狀態`、`主要產出單位`、`替代料策略摘要`、`來源建立方式` |
| 補充規格 | `doc/architecture/data/README.md` 配方治理要求 |

治理重點：

- 配方必須版本化，已被工單、生產或成本引用的版本不得直接覆寫。
- 配方版本不是單純說明文件，而是正式控制生產耗料與成本基礎的參照資料。
- 後續應再補配方明細 / BOM 結構、產出率、替代料規則與核定流程；本版先定義最低主檔骨架。

第一版最低可執行補強：

- 本版正式收斂配方版本的最低 key contract：`配方代碼 + 版本號`、對應產品或半成品、主要產出單位、生命週期欄位與替代料策略摘要。
- 已被工單、成本或 inventory 流程引用的版本，不得以覆寫方式修改；若內容改變到影響耗料或產出語意，必須建立新版本。
- 配方 BOM 明細、產出率公式、替代料明細與 maker-checker 流程仍留待後續獨立規格，不屬本版完成定義。

### 8. 出貨用品 / 包裝耗材

| 項目 | 定義 |
|------|------|
| 實體名稱 | 出貨用品 / 包裝耗材 |
| Owner module | Master Data |
| Authoritative source | 本文件第一版定義；後續應拆成獨立用品字典或主檔資料載體 |
| 正式 key | `用品代碼` |
| 主要 consumer | Procurement、Inventory、Order / Fulfillment |
| 最低必要欄位 | `用品代碼`、`用品名稱`、`用品類型`、`採購單位`、`庫存單位`、`庫存管理方式`、`是否出貨自動扣除`、`狀態`、`生效日`、`失效日`、`停用原因`、`來源建立方式` |
| 補充規格 | `doc/architecture/flows/shipping_supply_inventory_policy.md` |

治理重點：

- 紙箱、氣泡紙、膠帶、提袋等出貨用品都需要進 ERP，不能只停留在人工表單或口頭盤點。
- 這類用品屬正式主資料與庫存項目，但本輪明確定義 `是否出貨自動扣除 = 否`，出貨時不自動扣帳。
- 出貨用品的庫存維護以人工盤點與盤點 / 調整事件為主，後續若要半自動化，也必須先補正式流程與 RBAC。
- 內包裝包材不屬本類；若該用品會成為產品、分裝或包裝組成的一部分，應回到上方包材主檔治理，而不是落入人工盤點型耗材。

第一版最低可執行補強：

- 盤點週期以 `doc/architecture/flows/inventory_count_adjustment_and_negative_stock_policy.md` 為準：系統初期每週一次、穩定後每月一次、超過一個月未盤點需提醒。
- 最低調整原因字典以 `盤點差異`、`破損 / 報廢`、`遺失 / 盤虧`、`補貨建檔修正`、`規格切換停用` 為基線；若需新增原因，應走後續治理擴充。
- Phase 1 仍沿用 inventory-count 的過渡角色邊界處理這類人工盤點項目；正式倉管 / 履約角色細分留待後續 RBAC 擴版，不阻擋本版關帳。

- 目前蝦皮為唯一明確需要配置的渠道，但資料模型必須允許未來其他平台新增同類規則。
- 規則變更若會影響急單判定或切批提醒，必須保留生效日與停用追溯，不得覆寫既有歷史規則。
