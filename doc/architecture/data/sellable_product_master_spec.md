# 銷售商品主檔權威規格

更新日期：2026-03-26

Authoritative source：是

## 目的

本文件用於正式定義 Ivyhouse OP System 中 `銷售商品SKU_正式` 的 owner、欄位集合、生命週期與 consumer 邊界，補上先前由 `銷售商品組成對照表` 暫時兼任 owner 所留下的治理缺口。

本文件建立後：

1. `銷售商品SKU_正式` 的 owner 歸屬正式落到 Master Data
2. `銷售商品組成對照表` 的責任回到「組成與扣帳對照」
3. 後續訂單、出貨、報表、財務若要引用銷售商品 key，必須先依本規格建立正式主檔資料載體

## 範圍

本規格適用於所有可被銷售、接單、出貨、報表彙總、財務對帳引用的銷售商品，包含但不限於：

- 袋裝單品
- 盒裝單品
- 禮盒商品
- 綜合商品
- 咖啡 40 / 60 包盒裝
- 其他需要由內包裝與外包裝共同組成的銷售商品

本規格不直接涵蓋：

- 內包裝完成品主檔
- 外包裝材料主檔
- 生產／分裝／轉換規則主檔
- 訂單交易資料
- 發票、收款、核銷、對帳交易資料

## 實體定義

### 銷售商品

銷售商品是對外可被下單、販售、履約、開立發票與營收彙總的最小商業識別單位。

它是主資料，不是交易事實；因此：

- 不可在訂單、出貨或報表流程中臨時自由輸入成有效實體
- 必須有穩定的 `銷售商品SKU_正式`
- 必須有明確的啟用狀態與有效期間
- 不得把組成內容直接當成銷售商品主資料本身

### 銷售商品與內包裝完成品的差異

- 內包裝完成品是庫存、轉換、扣帳與禮盒內容物的正式主資料
- 銷售商品是市場上被販售與履約的正式主資料
- 一個銷售商品可對應一個或多個內包裝完成品
- 一個內包裝完成品也可能被多個銷售商品共用

### 銷售商品與組成表的差異

- 銷售商品主檔定義「這個商品是誰」
- 組成表定義「這個商品由哪些內包裝與外包裝組成」
- 主檔與組成表不可混為同一份長期權威資料

## Owner 決策

### 正式 owner module

- owner module：Master Data

### 決策理由

1. 專案規則已明定產品是主資料，不能在交易流程中臨時自由輸入成有效實體
2. 銷售商品將被訂單、出貨、報表與財務引用，屬跨模組 shared key
3. 若讓 Order / Fulfillment 或 Finance 持有 owner，會讓主資料與交易資料邊界混淆
4. 因此 `銷售商品SKU_正式` 必須回到 Master Data 維護

### 現況與過渡安排

- 現況：`project_maintainers/data/active/master-data/2026-03-26_銷售商品主檔_template.csv` 已建立，並已回填既有 `銷售商品SKU_正式`
- 現況：`project_maintainers/data/active/master-data/1-2026-03-24_銷售商品組成對照表_template.csv` 仍保有 `銷售商品SKU_正式` 欄位，作為組成 consumer 與對接鍵
- 過渡原則：在新增商品流程與下游 consumer 轉接完全文件化前，主檔與組成表需同步維護同一正式 SKU，且以主檔為先
- 限制：這不代表組成表是長期 owner；它只保留 consumer 與對接責任

## Shared Key 契約

### key 名稱

- `銷售商品SKU_正式`

### key 性質

- 類型：business key / shared domain key
- 用途：銷售、履約、報表、財務與人工對帳共用識別碼
- owner：Master Data

### key 規則

目前先採保守規則，不自行創造新編碼體系：

1. `銷售商品SKU_正式` 必須唯一且不可重複
2. `銷售商品SKU_正式` 一旦正式發布，不得就地改碼
3. 不得把組成版本直接編進 `銷售商品SKU_正式`
4. 不得把暫時包材變動、促銷文案或操作備註編進 `銷售商品SKU_正式`
5. 若未來需要制定正式編碼格式，必須另立 coding standard 或 ADR，不得在本規格未確認前自由發明格式

## 主檔欄位規格

### 最低必備欄位

正式銷售商品主檔資料載體至少應包含以下欄位：

| 欄位名 | 必填 | 說明 | 治理要求 |
|------|------|------|------|
| `銷售商品SKU_正式` | 是 | 銷售商品唯一正式識別碼 | 不可重複、不可就地改碼 |
| `銷售商品名稱` | 是 | 對外與內部共用的正式名稱 | 不可只存行銷文案，需可穩定辨識 |
| `銷售分類` | 是 | 例如夏威夷豆塔、鐵盒禮盒、濾掛咖啡 | 作為主資料分類維度 |
| `販售規格` | 是 | 例如 10入袋裝、禮盒、40包盒裝 | 屬商業識別的一部分 |
| `銷售商品類型` | 是 | 單品 / 組合 / 禮盒 / 盒裝等 | 不得混用自由文字 |
| `是否需要外包裝組裝` | 是 | 是否需要引用 PK 物料 | 應與組成表邏輯一致 |
| `對應組成規格代碼` 或等價欄位 | 是 | 指向銷售商品組成定義 | 不得只靠名稱比對 |
| `狀態` | 是 | 草稿 / 啟用 / 停用等 | 停用不等於刪除 |
| `生效日` | 是 | 主檔生效起始日 | 作為版本與歷史判定基礎 |
| `失效日` | 否 | 主檔失效日 | 若停用或改版時需填寫 |
| `備註` | 否 | 補充說明 | 不可承載正式規則 |

### 建議補充欄位

| 欄位名 | 用途 |
|------|------|
| `銷售商品短名稱` | 票據、報表、標籤或 UI 簡稱 |
| `顯示排序` | 後台清單與報表排序 |
| `主要銷售單位` | 訂單與發票顯示基準單位 |
| `停用原因` | 停用時的審計說明 |
| `來源建立方式` | 人工建立 / 匯入 / 遷移回填 |

### 狀態欄治理

- `狀態` 不可由自由文字取代，至少應區分 `草稿`、`啟用`、`停用` 或等價受控值
- `草稿` 代表主檔已建但尚未可供訂單、出貨、報表或財務正式引用
- `啟用` 代表主檔欄位、組成定義與必要包材引用都已完成，可作為正式 consumer 引用基準
- `停用` 代表該 SKU 保留歷史查詢與對帳能力，但不可再作為新交易的可售商品
- `停售` 若出現在名稱、分類或備註，只能表達業務語意，不得取代正式 `狀態`

## 與組成表的關係

### 責任分工

- 銷售商品主檔：定義銷售商品身分、狀態、有效期間與正式 owner key
- 銷售商品組成對照表：定義銷售商品實際扣哪些內包裝與外包裝

### 正式關聯方向

- 一筆銷售商品主檔，應對應至少一組有效的組成定義
- 一組組成定義，必須回指一筆存在且有效的銷售商品主檔

### 過渡期規則

- 在正式 `對應組成規格代碼` 尚未落地前，可暫以 `銷售商品SKU_正式` 作為主檔與組成表的對接鍵
- 但長期仍應補上獨立的組成規格識別碼或版本識別碼，避免主檔與組成版本綁死

## 版本化規則

### 銷售商品SKU 與版本分離

- `銷售商品SKU_正式` 不應承載版本語意
- 應版本化的是組成規則、包材配置、有效期間，必要時另以組成規格代碼或版本欄位管理

### 何時沿用同一 SKU

以下情況原則上沿用同一 `銷售商品SKU_正式`，但需建立新的組成版本或更新有效期間：

1. 內包裝配比調整，但商業身分不變
2. 外包裝材料替換，但對外販售商品仍被視為同一商品
3. 禮盒內容微調，但未改變對外識別與規格定義

### 何時必須新建 SKU

以下情況原則上應新建 `銷售商品SKU_正式`：

1. 銷售商品名稱、分類或販售規格已改變到足以構成新商品
2. 包數、入數、重量、盒型或禮盒層級改變，對消費者與財務辨識已屬不同商品
3. 同一商品需要在不同通路、稅別、履約邏輯下被當作不同銷售對象管理
4. 舊 SKU 已有交易事實，不適合被覆寫成新商品身份

## Consumer 邊界

### 訂單 / 出貨

- 訂單與出貨必須直接引用 `銷售商品SKU_正式`
- 訂單建立時可快照銷售商品名稱與販售規格，但不得只存快照文字、不存 SKU
- 出貨與履約若需展開扣帳，應透過主檔與組成定義解析，不得自由推導

### 報表 / 分析

- 報表應以 `銷售商品SKU_正式` 作為穩定維度鍵
- 報表可引用主檔中的分類、規格、狀態與有效期間，不應直接從組成表拼裝商品維度

### 財務 / 對帳

- 發票、營收、收款、核銷與對帳若涉及商品維度，應引用 `銷售商品SKU_正式`
- 成本歸集不可只看銷售商品名稱，必須透過 SKU 與組成／庫存異動鏈對接

### 組成表

- 組成表是 `銷售商品SKU_正式` 的 consumer，不是長期 owner
- 不得在主檔之外自行擴編新銷售商品 key

## 建立與變更流程

### 發布與同步原則

1. `銷售商品SKU_正式` 的新增、改碼、停用，一律先更新銷售商品主檔，再同步組成表
2. 組成表不得先行建立或擴編新的銷售商品 SKU，再回頭要求主檔補登
3. 主檔與組成表的同步應屬同一個變更單位，至少要共享同一生效語意或同一批次變更證據
4. 若主檔已改但組成表尚未同步完成，該 SKU 不得直接開放給訂單、出貨、報表或財務 consumer 使用

### 草稿轉正式

1. `草稿` 轉 `啟用` 前，必須已具備正式 `銷售商品SKU_正式`
2. 主檔最低必備欄位必須補齊，尤其是名稱、分類、規格、商品類型、組成對接、狀態與生效日
3. 對應組成定義必須存在，且若 `是否需要外包裝組裝 = 是`，相關外包裝材料引用必須有效，並符合 `OuterPackSkuContract` 的正式發布規則
4. 完成發布前，不得把 `草稿` SKU 當成正式可交易商品對下游擴散
5. 若商品已存在歷史交易，草稿轉正式不得改寫既有正式 SKU 身分

### 新增銷售商品

1. 先由 Master Data 定義銷售商品主檔欄位值與 `銷售商品SKU_正式`
2. 再建立或對接組成規格
3. 再允許訂單、出貨、報表或財務側引用

### 調整既有銷售商品

1. 先判斷是主檔屬性調整、組成版本調整，還是新商品建立
2. 不得直接用名稱替換掩蓋商品身份改變
3. 若影響訂單、出貨、財務或對帳結果，視為高風險變更

### 停用銷售商品

1. 不得物理刪除已有交易事實的銷售商品
2. 應以 `狀態` + `失效日` 停用
3. 停用後不得再被新訂單或新主檔關聯引用，除非有明確例外規則

### 停售與停用的關係

1. `停售` 是業務描述，不是正式生命週期狀態
2. 某商品即使在分類、名稱或備註上標示停售，仍應另以 `狀態 = 停用` 或等價受控值表達不可再販售
3. 若商品僅暫停販售但未正式退場，仍需明確定義是否允許恢復啟用，不得只靠備註傳達
4. 停售商品若保留歷史對帳與查詢需求，應保留原 SKU，不得以刪除或覆寫方式處理

## 遷移策略

### 目前起點

- 既有 `銷售商品SKU_正式` 已回填至銷售商品主檔，並同步存在於組成表作為對接鍵

### 正式落地後續步驟

1. 持續校驗主檔中的唯一 `銷售商品SKU_正式` 與組成表同步一致
2. 依本規格補齊主檔最低必備欄位的治理品質與發布規則
3. 盤點現有 `草稿` 商品哪些已具備轉 `啟用` 條件、哪些仍缺欄位或缺決策
4. 釐清現有停售商品應否正式轉為 `停用` 並補失效日 / 停用原因
5. 明確文件化「先進主檔、再同步組成表」的新增 / 改碼流程
6. 後續下游 consumer 一律直接引用主檔 owner，不得反向依賴組成表當 owner

### 遷移完成判定

需同時滿足以下條件才可宣稱 owner 落地完成：

1. 正式主檔資料載體已建立
2. 所有 `銷售商品SKU_正式` 均已在主檔註冊
3. 組成表不再作為新增商品的第一來源
4. 下游 consumer 已以主檔為準，而非直接依賴組成表作為 owner

## 目前結論

1. `銷售商品SKU_正式` 的 owner 已正式定義為 Master Data
2. 本文件是目前 owner 定義的權威來源
3. `銷售商品組成對照表` 目前仍是暫時操作來源，但不再是長期 owner 規格來源
4. 下一步應先盤點主檔 `草稿` / `停售` 商品的正式發布與停用決策，再推進下游 consumer 對接規則
