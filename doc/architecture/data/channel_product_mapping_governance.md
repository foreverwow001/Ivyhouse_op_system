# Channel Product Mapping Governance

更新日期：2026-03-29

Authoritative source：否（draft）

## 目的

本文件定義 Ivyhouse OP System 在 Phase 1 對「新增商品、平台商品名稱、映射規則與組合商品」的治理方式，回答一個核心問題：

> 未來若新增商品、新口味、新平台名稱或新的試吃組合，是否每次都需要進 VS Code 改 code？

本文件的正式答案是：

- 原則上不需要
- 應優先透過主資料、映射規則與組成規則治理
- 只有檔案版型或 parser grammar 改變時，才應進入工程改 code

## 分析結論

### 1. parser 與 mapping 必須分層治理

| 層次 | 負責內容 | 維護方式 |
|------|----------|----------|
| Parser layer | 讀檔、辨識平台、抽取欄位、解析固定語法 | 工程實作 |
| Mapping layer | 平台名稱別名、商品別名、SKU 對應、優先序 | 系統內規則維護 |
| Composition layer | 禮盒 / 試吃 / 組合商品的正式內容與 BOM | 系統內主資料維護 |

### 2. 新商品不應預設等於新 code

下列情況應由資料治理承接：

1. 新增銷售商品
2. 新增新口味或新規格
3. 平台商品名稱改寫
4. 增加平台別名與同義文字
5. 新增 `提袋加購`、咖啡品項或其他可銷售項
6. 新增 `試吃` 內的組合內容
7. 調整平台商品對應正式 SKU 的規則

## 應治理的資料物件

### 1. 銷售商品

用途：定義正式可販售 / 可彙總 / 可拆解的商品實體。

最低治理內容：

- 正式 SKU
- 商品名稱
- 商品類型
- 狀態
- 是否可被渠道 intake 引用
- 是否為組合 / 禮盒商品

### 2. 平台商品名稱別名

用途：承接不同平台對同一商品的命名差異。

最低治理內容：

- 渠道代碼
- 原始商品名稱或正規化名稱
- 原始規格別名
- 對應正式 SKU
- 映射優先序
- 生效 / 失效期間
- 狀態

治理原則：

- 這類資料應由系統維護，不應散落在 parser if/else
- 同一平台可存在多個歷史別名，但同時間只允許明確的有效規則集合

### 3. 組合 / BOM 定義

用途：承接禮盒、試吃、組合包與其他需展開的銷售結構。

最低治理內容：

- 組合商品正式 SKU
- 子品項 SKU
- 子品項倍率
- 生效 / 失效期間
- 是否允許 intake 時直接展開
- 狀態

治理原則：

- `試吃:` parser 只負責把句型切成「子品項 + 組數」
- 哪些子品項可成為正式需求，仍應引用正式商品與組成資料
- 新增試吃內容不應要求工程修改 parser，只要語法不變且商品已建檔

### 4. 特殊可銷售項目

用途：承接像 `提袋加購`、咖啡品項等不應再被直接過濾的項目。

治理原則：

- 若該項目屬正式銷售或正式履約需求，必須成為可治理商品或正式映射對象
- 不得再依賴 legacy analyzer 的硬編碼過濾規則直接略過
- 若暫時不進正式需求，應以「例外待人工處理」標記，而不是靜默丟棄

## 維護路徑

### 路徑 A：新增商品

適用情境：

- 新口味
- 新規格
- 新配方對外銷售
- 新增 `提袋加購`、咖啡等可銷售項

處理方式：

1. 建立正式銷售商品
2. 若為組合商品，建立組成 / BOM 定義
3. 補平台別名與映射規則
4. 經覆核後生效

### 路徑 B：平台名稱改動，但商品本體未變

適用情境：

- 平台把品名改成新文案
- 規格文案順序調整
- 新增平台特有別名

處理方式：

1. 不改 parser code
2. 補或停用映射規則
3. 保留版本與生效期間

### 路徑 C：新增 `試吃:` 組合內容

適用情境：

- `試吃:` 仍維持既有 grammar
- 但組內商品換成新 SKU 或新口味

處理方式：

1. parser 仍沿用既有 grammar
2. 先確保組內商品皆已存在正式商品主檔
3. 若需正式作為組合商品治理，再建立組成資料

### 路徑 D：真的需要改 code

只在下列情況啟動工程改動：

1. 渠道檔案版型改變
2. 欄位定位方式改變
3. 新增全新 grammar
4. 來源格式從 PDF/Excel 改成另一種媒介

## 權限與治理邊界

### 日常 intake 操作者可以做的事

1. 在單一批次內人工指定 SKU
2. 人工確認低信心映射結果
3. 將未映射項送入例外處理
4. 提報需要新增的商品或映射規則

### 日常 intake 操作者不應直接做的事

1. 直接發布永久映射規則
2. 直接新增正式主資料而不留審核
3. 直接覆寫既有規則歷史
4. 直接把單批修正視為永久規則變更

### 正式治理角色必須負責的事

1. 新商品核定
2. 映射規則新增 / 停用 / 改版
3. 組成 / BOM 正式發布
4. 歷史規則版本與生效日治理

## 最低必要欄位建議

### 平台商品映射規則

| 欄位 | 說明 |
|------|------|
| `映射規則代碼` | 正式規則識別 |
| `渠道代碼` | 平台來源 |
| `品名正規化模式` | 比對來源文字 |
| `規格正規化模式` | 比對規格文字，可空白 |
| `對應銷售商品SKU_正式` | 正式商品鍵 |
| `匹配類型` | 完全匹配 / 模式匹配 / 別名匹配 |
| `優先序` | 多規則衝突解決 |
| `生效日` | 正式啟用日 |
| `失效日` | 停用日 |
| `狀態` | 草稿 / 啟用 / 停用 |

### 平台商品別名

| 欄位 | 說明 |
|------|------|
| `渠道代碼` | 平台來源 |
| `原始品名字串` | 平台常見文字 |
| `正規化品名字串` | 去空白 / 去裝飾後字串 |
| `對應映射規則代碼` | 關聯正式規則 |
| `狀態` | 草稿 / 啟用 / 停用 |

## 例外回寫原則

例外處理流程完成後，應能把結果回寫為以下其中一類：

1. 單批接受，不升級為永久規則
2. 轉成新的映射規則草稿
3. 轉成新商品建立申請
4. 轉成新組成 / BOM 建立申請

不得出現「批次處理靠人工修完，但系統沒有留下可治理資料」的狀態。

## 驗收基準

### 驗收 1：新增商品不需直接改 code

- 只要 parser grammar 不變，新增商品或平台名稱應能在系統內完成治理

### 驗收 2：新 `試吃` 組合不需直接改 code

- 只要仍符合 `試吃:商品A*倍率+商品B*倍率 組數` grammar，即可由商品主檔與映射治理承接

### 驗收 3：永久規則與單批修正可區分

- 單批人工修正不得自動覆寫正式規則
