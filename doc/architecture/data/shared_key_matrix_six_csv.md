## 目的

本文件整理以下六張 CSV 之間的 shared key 關係，明確標示：

1. 哪一張表是 key owner
2. 哪些欄位是正式 shared key 欄位
3. 哪些表只是 consumer 或暫時提案來源
4. 哪些關係是直接引用，哪些只是間接依賴

適用範圍：

- `project_maintainers/data/active/master-data/1-2026-03-24_銷售商品組成對照表_template.csv`

## 六表欄位對接重點

### 1. 內包裝完成品SKU 是六表中最核心的 shared key

直接對接位置：

- 內包裝完成品主檔：`內包裝完成品SKU_正式`
- 銷售商品組成對照表：`對應內包裝成品SKU`
- 單位換算規則表：`適用對象SKU`
- 生產_分裝_轉換扣帳規則表：`輸入項目SKU`、`輸出項目SKU`
- SKU 編碼提案表：`建議SKU`

補充例外：

- 對於已核定為 `原料直接分裝 / no_inner_stage` 的 sellable family，`銷售商品組成對照表` 的 `對應內包裝成品SKU` 可暫承接原料主檔 `原料代碼`，作為組成輸入槽位。
- 這個例外只改變組成 consumer 槽位的可接受輸入，不改變 `內包裝完成品SKU` 與 `原料代碼` 各自的 owner 邊界。
- 正式規則以 `doc/architecture/data/shared_key_contract.md` 為準；本矩陣只負責標示六表對接現況。

### 2. 外包裝材料SKU 是第二核心 shared key

直接對接位置：

- 外包裝材料主檔：`外包裝材料SKU_正式`
- 銷售商品組成對照表：`外包裝材料SKU`
- 生產_分裝_轉換扣帳規則表：`輸入項目SKU`、`輸出項目SKU`（涉及 PK 規則時）
- SKU 編碼提案表：`建議SKU`

### 3. 單位換算規則表不是 owner，但它控制數量語意

重點：

- `適用對象SKU` 必須回指正式內包裝 SKU
- 換算率決定 RTW / RTP / RPK 規則的輸出數量是否可信
- 組成表中的 g / 片 / 包數量邏輯，必須與換算規則與正式 SKU 對齊
- 單位換算規則表雖不是 owner 主檔，但仍應以 `草稿` / `啟用` / `停用` 管理發布狀態
- 單位換算規則只有在引用的內包裝 SKU 已正式啟用、且換算率足以支撐現行規則與組成數量語意時，才可由 `草稿` 轉 `啟用`
- 若換算規則停用或改版，必須先確認受影響的轉換規則與組成數量語意已同步調整

### 4. 提案表屬提案來源，不可長期充當主檔

重點：

- 提案表只負責說明「新增 key 如何命名」
- 內包裝與外包裝一旦落地主檔，主檔才是正式權威來源
- 後續若提案表與主檔衝突，必須以主檔為準，再回頭修提案表紀錄

## 目前識別出的治理結論

1. `內包裝完成品SKU` 與 `外包裝材料SKU` 已形成可執行的 shared key contract
2. `規則代碼_正式` 已形成單表權威 key，但尚未被其他五表直接引用
3. `銷售商品SKU_正式` 的 owner 已由 `sellable_product_master_spec.md` 補齊，且正式主檔資料載體已建立
4. 提案表應保留為編碼決策痕跡，不應回升為正式 master owner

## 後續建議

1. 依 `sellable_product_master_spec.md` 補齊銷售商品主檔與組成表的同步治理規則，讓新增 / 改碼流程明確先進主檔再同步 consumer
2. 後續若引入訂單、出貨、報表或財務資料載體，應直接引用 `銷售商品SKU_正式`，不得再回退為名稱比對或組成表兼任 owner
